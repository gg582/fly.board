# Анализ подписи PQC (постквантовой криптографии)

## Обзор

fly.board заявляет о «квантово-стойких» подписях к постам. Этот документ объясняет, **как эта функция работает на самом деле**, и каковы её **реальные ограничения**.

## Как достигается PQC

### 1. Алгоритм: ML-DSA-65

- Согласно комментариям в `src/crypto/fly_crypto.h` и `README`.
- ML-DSA-65 — это **стандартизированная NIST решётчатая (lattice-based) схема подписи**.
- В отличие от RSA/ECDSA, которые опираются на факторизацию целых чисел и дискретное логарифмирование (уязвимые для алгоритма Шора на квантовом компьютере), решёточные задачи в настоящее время считаются устойчивыми к полиномиальным квантовым атакам.

### 2. Путь реализации: фреймворк CWIST → BoringSSL

- `fly_crypto.c` условно включает `<cwist/security/pqc/pqc_sig.h>`.
- Фреймворк CWIST встраивает BoringSSL, который содержит реализацию ML-DSA в `crypto/mldsa/`.
- Таким образом, fly.board **не реализует собственный PQC-алгоритм**, а просто **вызывает стандартный API ML-DSA, предоставляемый CWIST/BoringSSL**.

### 3. Подписываемые данные

- При создании или обновлении поста подписывается следующая строка:
  ```
  title + "\n" + content
  ```
- Подпись хранится в формате Base64 в таблице SQLite `posts`, колонка `pqc_signature`.
- При просмотре поста `fly_crypto_verify()` проверяет подпись, и в интерфейсе отображается значок «verified».

### 4. Управление ключами

- `fly_crypto_init()` генерирует **один глобальный ключ подписи** и хранит его в памяти.
- Ключ создаётся один раз при запуске приложения и освобождается при завершении работы через `fly_crypto_cleanup()`.

## Ограничения и предостережения

### 1. Аттестация сервера, а не аутентификация автора

- На сервере существует **только один ключ**, без разделения по пользователям.
- Поэтому подпись не доказывает, *какой* пользователь написал пост; она лишь подтверждает, что **сервер аттестует этот пост**.
- При компрометации сервера злоумышленник может использовать закрытый ключ сервера для подделки действительных подписей для произвольных постов.

### 2. Ловушка условной компиляции

- Все криптографические функции в `fly_crypto.c` обёрнуты в `#ifdef HAVE_PQC`.
- Если в среде сборки отсутствует `<cwist/security/pqc/pqc_sig.h>`, все функции становятся **no-op заглушками**: `fly_crypto_sign()` возвращает `false`, а `fly_crypto_verify()` всегда завершается неудачей.
- Поддержка PQC **не гарантируется**; она полностью зависит от наличия модуля CWIST PQC.

### 3. Ограниченная область подписи

- Подписываются только `title` и `content`.
- ID автора, временная метка, метаданные раздела и вложения **не включены в подписываемые данные**.
- Злоумышленник может изменить поле автора, оставив тело поста нетронутым, и подпись всё равно пройдёт проверку.

### 4. Транспортный уровень по-прежнему использует классическое шифрование

- fly.board использует HTTP/3 (QUIC) поверх TLS.
- Рукопожатие TLS и шифрование сессии **по-прежнему основаны на ECC/RSA**.
- Постквантовая подпись применяется только к *целостности тела поста*; она **не делает всё клиент-серверное общение квантово-безопасным**.

### 5. Отсутствие ротации и отзыва ключей

- Глобальный ключ генерируется один раз и не ротируется до перезапуска.
- Компрометация одного ключа одновременно разрушает доверие ко всем подписям.
- Нет интеграции с HSM, расписания ротации ключей или механизма отзыва.

## Заключение

Подпись PQC в fly.board использует **стандартизированный NIST ML-DSA-65 для защиты целостности тела поста**. Для движка блога это необычный и заслуживающий внимания эксперимент: это означает, что даже в эпоху квантовых компьютеров подделка поста остаётся обнаружимой.

Однако отсутствие **асимметричной аутентификации по автору, защиты метаданных, PQC на транспортном уровне и управления жизненным циклом ключей** делает маркетинговое утверждение о «сопротивляемости квантовым атакам» преувеличенным. Стоит рассматривать эту функцию как **демонстрацию PQC-подписи**, а не как полностью закалённый постквантовый слой безопасности.
