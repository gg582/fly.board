# HTTP/1.1·HTTP/2·HTTP/3 동시 지원 분석

## 개요

fly.board는 `cwist_app_use_https2()`와 `cwist_app_use_https3()`를 동시에 활성화하며, 모든 응답에 `Alt-Svc: h3=":%d"; ma=86400, h2=":%d"; ma=86400` 헤더를 포함한다. 이는 **HTTP/3를 우선하고, 불가능할 때 HTTP/2, 그리고 그마저도 안 될 때 HTTP/1.1로 자동 폴백(fallback)**하는 구조다.

## 왜 "성능을 포기하면서도" 세 가지를 모두 지원하는가

### 1. HTTP/3(QUIC) — 이상적이지만 환경 제약이 크다

- **장점**: 0-RTT 핸드셰이크, TCP Head-of-Line Blocking 완전 제거, 연결 마이그레이션(네트워크 전환 시에도 세션 유지).
- **성능 포기 요인**: UDP 기반 QUIC는 추가 암호화·패킷 재조립·혼잡 제어 오버헤드로 **HTTP/2 대비 CPU 사용량이 높다**. 또한 CWIST 낶부의 BoringSSL + ngtcp2/nghttp3 스택은 순수 HTTP/2 TCP 스택보다 더 많은 메모리와 코드 경로를 점유한다.
- **지원 이유**: 최신 브라우저(Chrome, Firefox, Safari)는 HTTP/3를 선호하지만, **엔터프라이즈 방화벽, 일부 NAT, 구형 프록시, 모바일 캐리어급 네트워크는 UDP를 차단하거나 QoS로 제한**한다. HTTP/3만 지원하면 이런 환경의 사용자는 접속 자체가 불가능해진다.

### 2. HTTP/2(TLS 1.3) — 현재의 "안정적 최대공약수"

- **장점**: 단일 TCP 연결 위의 멀티플렉싱으로 HTTP/1.1의 연결 낭비를 해결. 서버 푸시(선택적), 헤더 압축(HPACK).
- **성능 포기 요인**: HTTP/2는 HTTP/1.1보다 상태 기계가 복잡하고, 스트림 우선순위·흐름 제어(window update) 처리에 CPU 사이클을 소모한다. 벤치마크 상 HTTP/2는 이론상 HTTP/1.1 Keep-Alive보다 빠르지만, **fly.board의 벤치마크는 HTTP/2에서 측정**되었음을 명시하고 있다 — 이는 HTTP/3가 아직 모든 벤치마크 도구와 네트워크 환경에서 예측 가능한 결과를 내지 못함을 시사한다.
- **지원 이유**: HTTP/3가 막히는 환경에서의 **주력 대체재(primary fallback)**. 또한 `h2load`와 같은 성능 측정 도구가 HTTP/2를 표준으로 지원하며, 디버깅과 모니터링 생태계가 HTTP/3보다 성숙하다.

### 3. HTTP/1.1 — "마지막 복수수단"이자 호환성의 기반

- **장점**: 단순함. 텍스트 기반 프로토콜로 디버깅이 쉽고, 모든 로드밸런서·리버스 프록시·헬스체크 도구가 완벽하게 지원한다.
- **성능 포기 요인**: 단일 연결당 단일 요청-응답(파이프라이닝은 사실상 실패), 연결 수가 늘어나면 메모리와 파일 디스크립터 소모가 급증. HTTP/2의 멀티플렉싱 대비 **동시성 효율이 현저히 낮다**.
- **지원 이유**: 
  - **레거시 클라이언트**: 구형 CLI 도구, 일부 임베디드 기기, 오래된 크롤러는 HTTP/1.1만 이해한다.
  - **인프라 호환성**: 많은 로드밸런서와 WAF가 HTTP/2 이상을 종단에서 처리하지 못하고 HTTP/1.1으로 폴다운한다.
  - **헬스체크 및 모니터링**: `curl`, `wget`, Nagios/Zabbix 기반 체크는 여전히 HTTP/1.1이 가장 안정적이다.

## Alt-Svc 기반 우선순위 협상

```c
// src/handlers/handlers.c
snprintf(altsvc, sizeof(altsvc), 
    "h3=\":%d\"; ma=86400, h2=\":%d\"; ma=86400", 
    g_config.port, g_config.port);
```

- 클라이언트가 HTTP/2(또는 HTTP/1.1)로 첫 연결을 맺으면, 서버는 응답 헤더로 "같은 포트의 UDP로 HTTP/3도 가능하다"고 알린다.
- 클라이언트는 이 힌트를 캐시(`ma=86400`, 24시간)하고, **다음 요청부터 HTTP/3를 시도**한다.
- HTTP/3 실패 시 클라이언트 라이브러리(브라우저 등)는 자동으로 HTTP/2 → HTTP/1.1로 낙하한다.

## 성능을 포기하는 대신 얻는 것

| 포기하는 것 | 얻는 것 |
|-------------|---------|
| HTTP/3 단독 운영 시 이론상 최고의 지연 시간(latency) | **전 세계 사용자의 접근성**: UDP 차단 환경에서도 서비스 가능 |
| HTTP/2 단독 운영 시의 안정적 처리량 | **미래 대비**: HTTP/3가 표준이 되면 자동으로 최적 경로로 전환 |
| HTTP/1.1 단독 운영 시의 극도의 단순함과 낮은 메모리 사용 | **TASFA 대용량 파일 전송 최적화**: HTTP/2 멀티플렉싱 + HTTP/3 스트림이 청크 전송에 유리 |
| 서버 코드 복잡도 증가 (세 스택 동시 유지) | **운영 안정성**: 한 프로토콜 스택에 버그가 생겨도 다른 스택으로 자동 우회 |

## 결론

fly.board가 HTTP/1.1·HTTP/2·HTTP/3를 모두 지원하는 것은 "최신 프로토콜 하나만 고집해서 최고 성능을 내겠다"는 전략이 아니다. 오히려 **"어떤 클라이언트, 어떤 네트워크, 어떤 인프라 환경에서도 접속 가능하고, 가능한 한 최선의 프로토콜로 자동 전환되게 하겠다"**는 실용적 포석이다.

이는 블로그 엔진보다는 **다양한 단말과 네트워크 조건을 겪는 커뮤니티 플랫폼**에 적합한 설계 선택이다. HTTP/3의 고성능을 포기한 것이 아니라, HTTP/3가 불가능한 순간을 대비해 HTTP/2와 HTTP/1.1이라는 안전망을 유지하는 것이다.
