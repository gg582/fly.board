# TASFA 파일 전송 프로토콜 분석

## TASFA란 무엇인가

TASFA는 fly.board에서 사용하는 **HTTP/XHR 기반 파일 업로드/다운로드 프로토콜**이다. 단순히 파일을 주고받는 것을 넘어, 청크 단위 암호화·무결성 검증·불안정 네트워크 대응·DoS 방어를 통합한 프로토콜 계층이다.

## 핵심 구성 요소

### 1. 청크 단위 AES-256-GCM 스트리밍 암호화

- 업로드 세션마다 고유한 `stream_key`와 `iv_seed`를 생성한다.
- 각 청크는 `AES-256-GCM`으로 암호화되며, AAD(Additional Authenticated Data)로 `upload_id:chunk_index`를 사용한다.
- 태그(tag) 검증 실패 시 해당 청크는 즉시 폐기된다.
- 서버는 암호문을 받아 복호화한 뒤, 미리 할당된 임시 파일의 `chunk_index * chunk_size` 오프셋에 직접 쓴다.
- 이 암호화는 **세션 간 독립성**을 확보하여, 한 세션의 키가 노출되더라도 다른 세션에 영향을 주지 않는다. 다만 이는 TLS를 대체하는 것이 아니라 세션 레벨의 추가 무결성 검증 레이어다.

### 2. 6슬롯 HTP(Hexagonal Tortoise Problem) 로컬 복구 Lattice

- HTP는 **전송 프로토콜이 아니라 로컬 복구 힌트**다.
- 청크를 6개씩 그룹으로 묶고, 각 청크의 SHA-512 첫 8바이트를 64비트 정수 `H`로 해석한다.
- `raw_scalar = H % modulus_M`을 계산한 뒤, 그룹 내 세 직선 합의 불변식(`L1 == L2 == L3`)을 만족하도록 `v3`와 `v5`만 조정(balance)한다.
- 업로드 완료 시 서버가 모든 그룹의 불변식을 검증한다. 불일치 시 `400 Bad Request`를 반환한다.
- 이는 전체 파일 해시만으로는 알 수 없는 **청크 수준 치환·손상**을 조기에 탐지하는 가벼운 힌트다.

### 3. 파일 수준 순차 큐 (브라우저 연결 안정성)

- **한 번에 하나의 파일만 업로드**한다.
- 여러 파일을 선택하면 `FileUploadQueue`에 인큐되고, 현재 파일이 성공하거나 실패한 뒤에야 다음 파일이 진행된다.
- 브라우저의 origin당 6개 연결 제한을 존중하며, 연결 풀 고갈로 인한 교착을 방지한다.

### 4. 이진 비트맵으로 DoS 완화 및 상태 추적

- 업로드 상태는 `state.bin`에 **밀집 이진 비트맵**으로 저장된다(청크당 1바이트, `'0'`/`'1'`).
- 이미 `'1'`로 표시된 청크 인덱스로 재전송이 오면 서버가 거부한다. 이를 통해 동일 청크 재전송으로 디스크 I/O를 소모하는 DoS를 차단한다.
- `pwrite(..., 1, chunk_index)`로 O(1) 원자적 기록이 가능하며, 핫 패스에서 JSON 파싱이 전혀 없다.
- 다운로드 측에서도 비트맵을 유지하여, 이미 받은 청크는 스킵하고 `IndexedDB`에 캐싱한다. 재시도 폭풍(retry storm)에서 동일 데이터를 재요청하지 않는다.

### 5. 적응형 페이싱(Adaptive Pacing)

- 단순히 병렬도를 낮추는 대신, 링크 품질에 따라 청크 발송 간 `setTimeout` 지연(pacing)을 조절한다.
- | 링크 점수 | 업로드 페이싱 | 다운로드 페이싱 | Coalesce |
  |-----------|---------------|-----------------|----------|
  | >= 85     | 0 ms          | 0 ms            | 64       |
  | 65 – 84   | 15 ms         | 0 ms            | 48       |
  | 45 – 64   | 15 ms         | 10 ms           | 32       |
  | < 45      | 35 ms         | 30 ms           | 16       |
- 고손실 링크에서 TCP 윈도우 붕괴를 막고, 버퍼 블로트(버퍼 팽창)로 인한 멈춤을 방지한다.
- 링크 점수가 45 미만이면 타임아웃을 늘리고(최대 12초), 재시도 백오프 계수를 1.6에서 1.3으로 낮춘다.

### 6. 세션 강화

- 업로드 ID와 토큰은 각각 16바이트·24바이트 무작위 16진수 문자열이다.
- `flock`을 사용하여 동시 요청으로부터 비트맵 갱신 경쟁을 방지한다.
- 다운로드 세션은 TTL(`TASFA_DOWNLOAD_TTL = 86400`)로 제한되며, 오래된 세션은 자동 폐기된다.

## RPS를 희생할 가치가 있는가

> **참고**: 아래 RPS 트레이드오프는 `/file/upload`, `/file/download` 등 **파일 전송 엔드포인트에만 해당**한다. 벤치마크에서 측정된 홈·게시판 목록 등의 페이지 RPS는 TASFA와 무관하다.

TASFA는 단순 정적 파일 서빙에 비해 분명한 **RPS(Request Per Second) 손해**를 낳는다. 그러나 이 손해는 다음 가치를 얻기 위한 의도된 트레이드오프다.

| 희생하는 것 | 얻는 것 |
|-------------|---------|
| 핫 패스에서의 추가 암호화/복호화 오버헤드 | **청크 단위 무결성**: 태그 불일치 시 즉시 탐지, 변조된 청크가 디스크에 기록되지 않음 |
| HTP scalar 계산 및 그룹 검증에 소모되는 CPU | **조기 손상 탐지**: 전체 파일이 모두 전송된 뒤에야 알 수 있는 오류를 그룹 단위에서 미리 검출 |
| 비트맵 파일 I/O와 `flock` 세션 락 | **DoS 방어**: 동일 청크 재전송 남용, 트렁케이트 파일 공격(truncated-file attack) 차단 |
| 순차 큐로 인한 다중 파일 동시 업로드 불가 | **브라우저 연결 안정성**: origin당 6연결 제한 내에서 교착 없이 안정적 전송 |
| Pacing으로 인한 낮은 청크 발송률 | **불안정 링크 복원력**: 고손실·고지연 환경에서도 TCP 윈도우 붕괴 없이 지속 전송 |
| IndexedDB 캐싱과 세션 상태 유지로 인한 메모리·디스크 사용 | **재시도 효율화**: 네트워크 단절 후 재연결 시 이미 받은 청크는 재요청하지 않음 |

### 결론

TASFA는 "가장 빠른 파일 전송"을 추구하지 않는다. 대신 **파일이 불안정한 네트워크와 악의적 클라이언트 환경에서도 완전하고 무결한 상태로 도달할 수 있도록** 설계되었다.

- **업로드 관점**: 세션별 암호화된 청크 스트림 + HTP 힌트 + 비트맵 DoS 방어 + 순차 큐 = 안정적이고 무결한 업로드.
- **다운로드 관점**: 핸드셰이크 기반 세션 + 비트맵 추적 + IndexedDB 캐싱 + 적응형 페이싱 = 끊김 없는 다운로드.

fly.board가 단순 블로그라면 TASFA는 과하지만, **첨부 파일 기반 커뮤니티**에서는 RPS 손해를 감수할 만한 명확한 기술적 타당성을 가진다.
