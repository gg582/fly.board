# TASFA ファイル転送プロトコル解析

## TASFA とは何か

TASFA は fly.board で使用される **HTTP/XHR ベースのファイルアップロード/ダウンロードプロトコル** である。単なるファイル送受信を超え、チャンク単位の暗号化・完全性検証・不安定ネットワーク対応・DoS 対策を統合したプロトコル層である。

## 核心構成要素

### 1. チャンク単位 AES-256-GCM ストリーミング暗号化

- アップロードセッションごとに固有の `stream_key` と `iv_seed` を生成する。
- 各チャンクは `AES-256-GCM` で暗号化され、AAD（追加認証データ）として `upload_id:chunk_index` を使用する。
- タグ検証に失敗したチャンクは即座に廃棄される。
- サーバーは暗号文を受信して復号し、事前に割り当てられた一時ファイルの `chunk_index * chunk_size` オフセットに直接書き込む。
- この暗号化は **セッション間の独立性** を確保し、あるセッションの鍵が漏洩しても他のセッションに影響を与えない。ただしこれは TLS に代わるものではなく、セッションレベルの追加完全性検証レイヤーである。

### 2. 6 スロット HTP（Hexagonal Tortoise Problem）ローカル修復 Lattice

- HTP は **トランスポートプロトコルではなく、ローカル修復ヒント** である。
- チャンクを 6 個ずつグループ化し、各チャンクの SHA-512 先頭 8 バイトを 64 ビット整数 `H` として解釈する。
- `raw_scalar = H % modulus_M` を計算した後、グループ内の 3 直線和の不変式（`L1 == L2 == L3`）を満たすように `v3` と `v5` のみを調整（balance）する。
- アップロード完了時にサーバーがすべてのグループの不変式を検証する。不一致の場合は `400 Bad Request` を返す。
- これはファイル全体ハッシュだけでは判明しない **チャンクレベルの置換・破損** を早期に検出する軽量なヒントである。

### 3. ファイルレベル順次キュー（ブラウザ接続の安定性）

- **一度に 1 つのファイルのみアップロード** する。
- 複数ファイルを選択すると `FileUploadQueue` にエンキューされ、現在のファイルが成功または失敗した後に次のファイルが進行する。
- ブラウザの origin あたり 6 接続制限を尊重し、接続プール枯渇によるデッドロックを防ぐ。

### 4. バイナリビットマップによる DoS 緩和と状態追跡

- アップロード状態は `state.bin` に **密なバイナリビットマップ** として保存される（チャンクあたり 1 バイト、`'0'`/`'1'`）。
- 既に `'1'` とマークされたチャンクインデックスへの再送はサーバーが拒否する。これにより、同一チャンク再送でディスク I/O を消費させる DoS をブロックする。
- `pwrite(..., 1, chunk_index)` により O(1) 原子書き込みが可能であり、ホットパスでの JSON パースは一切ない。

### 5. セッション強化

- アップロード ID とトークンはそれぞれ 16 バイト・24 バイトのランダム 16 進数文字列である。
- `flock` を使用して、同時リクエストからのビットマップ更新競合を防ぐ。
- ダウンロードセッションは TTL（`TASFA_DOWNLOAD_TTL = 86400`）で制限され、古いセッションは自動廃棄される。

## RPS の犠牲に価値はあるか

> **注**: 以下の RPS トレードオフは `/file/upload`、`/file/download` などの**ファイル転送エンドポイントにのみ該当**する。ベンチマークで測定されたホーム・掲示板一覧などのページ RPS は TASFA に影響を受けない。

TASFA は単純な静的ファイル配信と比較して明確な **RPS（秒間リクエスト数）の損失**をもたらす。しかし、この損失は以下の価値を得るための意図されたトレードオフである。

| 犠牲にするもの | 得られるもの |
|-------------|---------|
| ホットパスでの追加暗号化/復号オーバーヘッド | **チャンク単位の完全性**: タグ不一致時は即時検出され、改ざんチャンクはディスクに書き込まれない |
| HTP スカラー計算およびグループ検証に消費される CPU | **早期破損検出**: ファイル全体が送信されてからでないと判明しないエラーを、グループ単位で事前に検出 |
| ビットマップファイル I/O と `flock` セッションロック | **DoS 対策**: 同一チャンク再送の乱用、トランケートファイル攻撃のブロック |
| 順次キューによる複数ファイル同時アップロードの不可 | **ブラウザ接続の安定性**: origin あたり 6 接続制限内でデッドロックなしの安定転送 |
| セッション状態保持によるメモリ・ディスク使用 | **リトリー効率化**: ネットワーク断絶後の再接続時、サーバーは未受信チャンクを正確に把握 |

### 結論

TASFA は「最速のファイル転送」を追求しない。代わりに、**ファイルが不安定なネットワークや悪意のあるクライアント環境でも、完全かつ無欠の状態で到達できるよう**設計されている。

- **アップロード観点**: セッション単位の暗号化チャンクストリーム + HTP ヒント + ビットマップ DoS 防御 + 順次キュー = 安定した無欠なアップロード。
- **ダウンロード観点**: ハンドシェイクベースセッション + ビットマップ追跡 + 適応型 coalescing = 途切れないダウンロード。

単純なブログであれば TASFA は過剰だが、**添付ファイル重視のコミュニティ**では RPS 損失を許容するに値する明確な技術的根拠を持つ。
