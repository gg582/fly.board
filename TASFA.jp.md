# TASFA

TASFA は本プロジェクトでファイルのアップロードとダウンロードに使用される転送プロトコルである。

単純な HTTP/XHR の上に構築され、チャンク単位の暗号化、ファイルレベルの順次キュー、および **サーバー権限の 6 スロット HTP（Hexagonal Tortoise Problem）復元格子（recovery lattice）** を追加する。

## ルート

アップロード:
- `POST /file/upload/init`
- `POST /file/upload/status`
- `POST /file/upload/renegotiate`
- `POST /file/upload`
- `POST /file/upload/complete`
- `POST /file/upload/cancel`

ダウンロード:
- `GET /file/download/:id/handshake`
- `GET /file/download/:id/chunk/:chunk_index`
- `GET /file/download/:id`
- `GET /assets/tasfa/img/:filename/handshake`
- `GET /assets/tasfa/img/:filename/chunk/:chunk_index`
- `GET /assets/tasfa/uploads/:filename/handshake`
- `GET /assets/tasfa/uploads/:filename/chunk/:chunk_index`

## アップロードプロトコル

ブラウザはまずアップロードセッションを交渉し、TASFA ヘッダーと共に **チャンク**（デフォルト `16 MiB`、モバイル `8 MiB`）を送信する:

- `X-TASFA-Upload-ID`
- `X-TASFA-Upload-Token`
- `X-TASFA-Chunk-Index`
- `X-TASFA-Hash-Tag`
- `X-TASFA-Raw-Scalar`
- `X-TASFA-Magic-Scalar`

サーバーは各チャンクを事前に割り当てられた一時ファイルの `chunk_index * chunk_size` オフセットに直接書き込む。トランスポートブロックはもはや存在しない。

通常のチャンクが繰り返し失敗する場合、ブラウザは `X-TASFA-Stream-Mode: aes-256-gcm` フォールバックリクエストでその失敗チャンクを送信する。サーバーは暗号文と GCM 認証タグを受け入れ、フォールバックチャンクは適応型並列ウィンドウ内に留まる。フォールバックリクエストにも同じ HTP ハッシュタグとバランススカラーヘッダーが含まれる。

### 剰余（最後の部分チャンク）

ファイルサイズがチャンクサイズの倍数でない場合、最後のチャンクは **剰余（remainder）** である。正確なバイト範囲を単一の blob として送信し、サーバーは正しいオフセットに書き込む。パディングや分割は行わない。

### セッション初期化レスポンス

`init` エンドポイントは以下を含むレスポンスを返す:

- `chunk_size` — 交渉されたチャンクサイズ
- `current_parallel_chunks` — サーバーが承認した現在のアップロードウィンドウ
- `max_parallel_chunks` — クライアントが同時にアップロードできるチャンク数
- `dispatch_pacing_ms` — 測定されたリンクが悪い場合のみ適用される小さな送信間隔
- `upload_secret` — 暗号化ストリーム検証用の副次的サーバーシークレット
- `stream_key_hex`, `stream_iv_seed_hex`, `stream_mode` — セッション暗号化キー（`aes-256-gcm`）
- `modulus_M` — このセッションで使用される HTP モジュラス
- `group_count` — 完全な 6 スロット HTP グループの数
- `client_stripes` — クライアントワーカースケジューラ用の固定値 `32`

### アップロードチャンクレスポンス

チャンクが受け入れられた場合、サーバーは `204 No Content` と以下のヘッダーで応答する:

- `X-TASFA-Accepted: 1`
- `X-TASFA-Chunk-Complete: 1`

チャンクインデックスが既に `state.bin` で完了としてマークされており、かつそのインデックスがサーバーの `retry_targets` にない場合、サーバーは同じヘッダーで `204` を返すが、チャンクボディは破棄される。

## HTP サーバー権限復元格子

HTP は **トランスポートプロトコルでも暗号学的証明でもない**。チャンクレベルで破損の可能性をランク付けし、クライアントがファイル全体ではなく高リスクの疑いチャンクのみを再送信するように支援するサーバー側エンジンである。

### チャンクグループ化

チャンクは連続した 6 要素の頂点にグループ化される:

```
グループ g: [ v0 , v1 , v2 , v3 , v4 , v5 ]
              chunk g*6+0  ...  g*6+5
```

最後のグループは部分的である可能性があるが、**不完全なグループは絶対にパディングされず、HTP 検証から完全に除外される**。ゼロパディングは人工的なトポロジーを注入するため禁止される。

### 生スカラー（raw scalar）

各チャンクに対し、クライアントは平文チャンクの SHA-512 を計算し、先頭 8 バイトをビッグエンディアン符号なし 64 ビット整数 `H` として解釈し、以下を導出する:

```
raw_scalar = H % modulus_M
```

### 魔法線不変式（magic line invariant）

完全なグループに対し、3 本の線を定義する:

```
L1 = v0 + v1 + v2   (mod M)
L2 = v2 + v3 + v4   (mod M)
L3 = v4 + v5 + v0   (mod M)
```

不変式は `L1 == L2 == L3` を要求する。生スカラーがこれを満たさない場合、クライアントは **v3 と v5 のみを調整**してバランスを取る:

```
delta2 = (L1 - L2) mod M
delta3 = (L1 - L3) mod M

v3_balanced = (v3_raw + delta2) mod M
v5_balanced = (v5_raw + delta3) mod M
```

他のすべての頂点は生スカラーを維持する。クライアントは両方を送信する:

- `X-TASFA-Raw-Scalar` — 未変更の `raw_scalar`
- `X-TASFA-Magic-Scalar` — バランス値 (`v3_balanced` または `v5_balanced`、それ以外は raw と同じ)

サーバーは両方のスカラーを `htp.bin` に個別に保存するため、人工的なバランス制約とは無関係に元のトポロジーを参照できる。

### なぜ v3 と v5 だけか？

六角格子は 2 つの自由度を持つ。`v0,v1,v2,v4` を固定し `v3,v5` を調整することで、3 本の線方程式を一意に満たしながら、デルタを最小化しグループ内で局所的に保つことができる。

## サーバー権限 HTP 復元

**クライアントは無知な再送信エージェント（dumb retransmission agent）である。** クライアントは修復代数を計算せず、コスト閾値を評価せず、疑いランク付けを導出しない。すべてはサーバー側のみに存在する。

### サーバー検証フロー

`POST /file/upload/complete` の間、サーバーは以下を実行する:

1. `htp.bin` からチャンク別レコード（ハッシュタグ、生スカラー、バランススカラー）をすべてロードする。
2. **完全な 6 スロットグループのみ** を検証する（部分グループはスキップする）。
3. 6 スロットグループ内でスカラー順序が置換（out-of-order）されている場合、サーバーは 720 通りの全順列を探索して不変式を復元する順列を見つけ、`htp.bin` レコードを並び替えて修復し、再送信なしで通過させる。
4. 失敗したグループごとに、各スロットが参加する線方程式を分析して **疑いスコア（suspicion scores）** を計算する。

### 疑い信頼度スコア（グループ別）

失敗したグループに対し、サーバーは各スロットを 3 本の線方程式と比較して評価する:

| スロット | 方程式 |
|---------|--------|
| v0      | L1, L3 |
| v1      | L1     |
| v2      | L1, L2 |
| v3      | L2     |
| v4      | L2, L3 |
| v5      | L3     |

各スロットの疑いスコアはトポロジーからのみ決定論的に導出される:

```
score = in_fail / total_fail
```

ここで `in_fail` はそのスロットが参加する失敗線方程式の数、`total_fail` はそのグループの総失敗方程式数である。任意の信頼度定数は使用しない。

合格方程式のみに登場するスロットは疑いリストから **除外** される。

スコアはすべての失敗グループにわたって集約され、1 つのチャンクが複数のグループに現れる場合最大スコアが維持される。

### 修復コスト閾値

サーバーは収縮を要求する前に、収縮が直接再試行より安価かどうかを評価する:

```
repair_worthwhile(疑い数, 全体チャンク数, チャンクサイズ, rtt_ms):
    if 疑い数 < 3                → false  （トポロジーに意味なし）
    retry_cost  = 疑い数 * チャンクサイズ * rtt_factor(rtt_ms)
    repair_cost = メタデータバイト + サーバーCPUコスト + 追加RTTコスト
    return retry_cost > repair_cost
```

抽象的コストモデルは、クライアントが再送信するバイトとサーバー側分析オーバーヘッドを比較する。大きなチャンクや高遅延は収縮をより魅力的にし、多数の小さな疑いチャンクは直接再試行をより安価にする。具体的な数値はサーバー側実装の詳細であり、プロトコル定数ではない。

閾値が修復を拒否した場合、サーバーは **すべての** 疑いチャンクを再試行対象として含む `needs_retry` を返す。クライアントは通常のアップロードエンドポイントを介して再送信する。

### サーバー側再帰的収縮

修復が価値ある場合、サーバーは **グループレベル収縮** を実行する: 各元の完全な 6 スロットグループは自身の **不変式シグネチャ** をエンコードした単一のスカラーに縮小される。サーバーはグループの 3 本の線値 `L1, L2, L3` を計算し、残差 `r12 = (L1-L2) mod M` と `r23 = (L2-L3) mod M` を導出して、グループ集約値を `(r12 * r23) mod M` と設定する。通過グループは `r12 = r23 = 0` なので集約値は `0` となり、失敗グループは線の不整合のトポロジーを保持する非ゼロの決定論的シグネチャを受け取る。これらのグループ集約値がより高いレベルの HTP 格子の頂点となる。連続した 6 つのグループ集約値がレベル 1 スーパーグループを形成し、同じ線不変式が再評価される:

- レベル 1 スーパーグループが通過すると、基盤となるレベル 0 グループの疑いチャンクがクリアされる（グループレベルで失敗パターンが一致する）。
- レベル 1 スーパーグループが失敗すると、基盤となるレベル 0 グループの疑いチャンクが保持される。
- 収縮が疑い集合を縮小した場合（より少ないチャンク）、サーバーは縮小された対象を保存し、縮小リストで `needs_retry` を返す。
- 収縮が集合を縮小できなかった場合、サーバーは元の疑いチャンクの直接再試行にフォールバックする。
- 収縮レベルはセッションメタデータに増加され、クライアントが診断を報告できるようにする。

クライアントは収縮グループを見たり計算したりしない。`retry_targets` のみを受信する。

### 再送信受け入れ

クライアントが既に受信済みとしてマークされたチャンクを再送信する際、通常のアップロードエンドポイントは **そのチャンクインデックスが現在サーバーの `retry_targets` リストにある場合のみ** 再送信を受け入れる。再送信チャンクが保存された後、サーバーは `retry_targets` からこれを削除する。

### プロトコル可視修復レスポンス

HTP 失敗時にサーバーが修復または再試行を決定すると、`complete` エンドポイントは `409` と共に以下を返す:

- `htp_status`: `"needs_retry"`
- `retry_targets`: 再送信するチャンクインデックスの配列（疑いスコア降順）
- `suspicion_scores`: `{chunk_index, score}` オブジェクトの配列
- `contraction_level`: 適用されたサーバー側収縮通過回数
- `retry_reason`: 人間が読める説明（例: `"htp group inconsistency detected"`）

コスト閾値が直接再試行の方が安価だと判断した場合、`retry_targets` は完全な疑いリストを含み、`contraction_level` は `0` のままである。

サーバーが収縮により疑いを縮小できた場合、`retry_targets` は縮小リストを含み、`contraction_level` が増加する。

すべての疑いチャンクが再送信され成功裏に検証されると、次回の `complete` 呼び出しは SHA-256 最終化へ進む。

## ファイルレベル順次キュー

**一度に 1 つのファイルのみをアップロードする。** 複数のファイルを選択した場合:

1. 各ファイルは独自のアセット、プレビューカード、HTP セッションを持つ。
2. ファイルは `FileUploadQueue` にエンキューされる。
3. アクティブなファイルが完了（成功または失敗）すると、キューは自動的に次のファイルに進む。
4. 一括「キューに入れたファイルをアップロード」ボタンは常に有効であり、クリックするとすべての保留中ファイルをエンキューしてポンプを開始する。

これによりブラウザの接続プール枯渇が防止され、ストール検出が安定的に維持される。

## ランタイム設定

- アップロードチャンクサイズ: デスクトップ `16 MiB`、モバイル `8 MiB`
- 適応型アップロードチャンクサイズヒント: 最小 `8 MiB`、最大デスクトップ `32 MiB` / モバイル `16 MiB`
- ダウンロードチャンクサイズ: デスクトップ `8 MiB`、モバイル `4 MiB`、クライアントが大きなセッションをヒントした場合最大 `32 MiB`
- デフォルトブラウザアップロード並列性: `16`
- 最大ブラウザアップロード並列性: `blog.settings` の `max_upload_parallel_chunks`、上限 `40`
- 最大同時アップロードセッション: `blog.settings` の `max_total_parallel_uploads`、上限 `64`
- 最大アップロードサイズ: `blog.settings` の `max_upload_size`
- 最大ブラウザダウンロードセッション: サーバー定義、現在セッションあたり最大 `48` チャンクリクエスト
- ダウンロード結合（span グループサイズ）: 良好なリンクで最大 `16` チャンク
- アップロード xhr タイムアウト: 少なくとも `180 s`
- アップロードセッション fetch タイムアウト: `30 s`

ブラウザの origin あたり HTTP 接続制限はワーカープールによって自然に尊重される。

### クライアント適応

アップロードクライアントはチャンク完了時間、再試行、タイムアウト、および利用可能な場合はNetwork Information APIのヒントを測定する。これらの入力を `/file/upload/init` と `/file/upload/renegotiate` に送信し、サーバーは現在の並列ウィンドウと最大ウィンドウを返す。クリーンに完了するチャンクが続くと、アクティブウィンドウを協議された最大値まで `*1.15` で増やし、一時的な失敗は `*0.85` で減らす。AES-GCM フォールバックも同じ適応型ウィンドウ内で実行される。

チャンクごとの絶対タイムアウトにより、一部分のデータだけを受信して接続が閉じないスタルを防ぐ。watchdogが停滞したアップロードを検出すると（タイムアウト内にチャンク完了なし）、クライアントは**攻撃的リスタート(aggressive restart)**を実行する：`targetParallel` を `maxParallel` にリセットし、`dispatchPacingMs` を `0` に設定してからサーバーのビットマップからアップロードを再開する。学習されたチャンクサイズのヒントは**セッションローカル**に保持され、`localStorage` には即座に保存されず、次のTASFAセッションにのみ適用される。

ダウンロードも同じ高スループット偏向を使用する。ハンドシェイクにはクライアントの優先チャンクサイズヒントが含まれ、アクティブなダウンロードは成功したチャンクグループの後で `span` と並列度を増やす。短い応答、タイムアウト、ネットワークエラーはチャンクインデックス単位で再キューに入れられ、同じチャンクが高い再試行予算を使い果たすまでは、全体のダウンロードを失敗で終わらせない。

### 後半部RTT予測（ラグランジュ外挿）

大きなファイル（チャンク数の多いセッション）では、サーバーはクライアントから報告されたチャンクごとのRTTサンプルを最大8個まで蓄積する。サンプルが3個以上集まると、ラグランジュ多項式外挿で最後のチャンクインデックスまでのRTTを推定し、残りのチャンク数を掛けて推定残り時間を算出する。

- アップロード: 各チャンク転送完了後、クライアントは次のチャンク要求に `X-TASFA-Chunk-RTT` ヘッダー（ミリ秒）を付加できる。
- ダウンロード: クライアントは前のチャンクのRTTを `chunk_rtt_ms` クエリパラメータで別途報告できる。

サーバーは状態応答JSONに `predicted_remaining_ms` を追加し、ダウンロードチャンク応答には `X-TASFA-Predicted-Remaining-Ms` ヘッダーを含める。数値演算は意図的に軽量に保つ: サンプル上限8個、O(n²)の単純ラグランジュ、予測値は30秒上限でクリッピングする。

### Wynn加速予備防衛

TASFAは平文バイトを予測したり、予測されたチャンク内容を信頼したりしない。予測対象はクライアントが観測した転送品質数列、すなわち完了時間、スループット、再試行、タイムアウト、短い応答のみである。クライアントは最新の品質サンプルに対し、Wynn epsilon変換のAitken `Delta^2` 形態を適用する：

```
S_hat = S0 - ((S1 - S0)^2 / (S2 - 2*S1 + S0))
```

`S_hat` は `[0, 1]` の範囲に制限され、次のチャンクの防衛信号としてのみ使用される。予測品質が高い場合は通常経路を維持し、適応型ウィンドウを拡大する。中程度以下に低い場合は次のチャンクを `guarded` とマークし、送信前にアクティブ並列ウィンドウを1つ減らす。非常に低い場合は次のアップロードチャンクを並列処理可能なAES-GCMフォールバックに先行切り替えたり、ダウンロードの `span` と並列度を次のrangeグループ要求前に減らしたりする。これは暗号学的オラクルではなく、前向きの輻輳/失敗防衛である。

## ストレージモデル

各アップロードセッションは 1 つの一時ファイルを事前に割り当てる:

- 一時ファイル: `data/tasfa/uploads/<upload_id>/upload.bin.part`
- メタデータ: `data/tasfa/uploads/<upload_id>/meta.json`
- 高速バイナリメタ: `data/tasfa/uploads/<upload_id>/meta.bin`
- 状態: `data/tasfa/uploads/<upload_id>/state.json`, `data/tasfa/uploads/<upload_id>/state.bin`
- HTP レベル 0: `data/tasfa/uploads/<upload_id>/htp.bin`
- セッションロック: `data/tasfa/uploads/<upload_id>/session.lock`

サーバーはもはや `blocks.bin` や `chunk_counts.bin` を維持しない。チャンク完了は単一のビットマップ書き込みである。

セッションメタデータはまた以下の頂点別配列を保存する:

- `hash_tags` — チャンクごとの SHA-512 16 進文字列配列
- `magic_scalars` — チャンクごとのバランススカラー配列
- `htp_retry_targets` — 現在サーバーが発行した再試行対象リスト
- `htp_suspicion_scores` — 現在の疑いランク付け
- `htp_contraction_level` — 適用されたサーバー側収縮通過回数

これらはすべてのチャンクアップロード時に更新され、完了時に検証される。

## 削除 PIN

完了したアップロードは使い捨ての削除 PIN を受け取る。平文 PIN は 12 文字の 16 進数文字列（6 バイトのランダム）であり、一度だけ返され、ハッシュのみが保存される。

## ダウンロードプロトコル

1. クライアントは `GET /.../handshake` を通じてハンドシェイクをリクエストする。
2. サーバーは `session_id`、`session_token`、`chunk_size`、`chunk_count`、並列性ヒント、およびセッション暗号化キー（`stream_key_hex`、`stream_iv_seed_hex`）を返す。
3. クライアントは適応的な `span=...` でチャンクグループを取得する。セッションキーがある場合、すべてのチャンクは **AES-256-GCM** で暗号化される。
4. クライアントはブラウザで **Web Crypto API** とセッションキーを使用してチャンクを復号する。
5. ブラウザはレスポンスを1つの連続したバッファに組み立てる。

## ビットマップによる DoS 緩和

アップロードとダウンロードの状態は両方とも **密なバイナリビットマップ**（チャンクあたり 1 バイト、`'0'` / `'1'`）で追跡される。

### アップロード側

- サーバーは `state.bin` で既に `'1'` とマークされたチャンクインデックスの再送信を拒否する。**ただし、そのチャンクがサーバーの再試行対象リストに明示的にある場合は除く**。攻撃者は任意のチャンクを再生してディスク I/O を消費できない。
- `state.bin` は `pwrite(..., 1, chunk_index)` で O(1) 原子書き込みが可能であり、ホットパスでの JSON パースは一切ない。
- `complete` ハンドラーはビットマップを再び開いて設定されたビットを数え、`received_chunks == chunk_count` になるまで最終確定を拒否する。これにより切り詰めファイル攻撃が防止される。

### セッション強化

- アップロード ID とトークンはそれぞれ 16 バイト・24 バイトのランダム 16 進数文字列である。
- セッションロック（`session.lock` への `flock`）は同時リクエストからの競合するビットマップ更新を防ぐ。

## ワーカースケジューラ

サーバーは固定ラウンドロビンワーカースケジューラを使用する。ワーカー数は CPU コア数と等しい（最小 2、最大 64）。同じ `upload_id` に属するチャンクは、キャッシュ局所性と順序のために常に同じワーカーにディスパッチされる。ジョブはワーカーごとのキューを介して送信され、条件変数でシグナルされる。

## 非同期最終化

`POST /file/upload/complete` は非同期に処理される。最初の呼び出しは `202 Accepted` と `{"processing": true}` を返し、バックグラウンド最終化ワーカーを開始する。後続の呼び出しは最終化キャッシュをポーリングし、ワーカーが完了するとキャッシュされたステータスとボディが即座に返される。これにより、長時間実行されるファイナライズ処理が HTTP 接続をブロックしないようにする。

## 自己レビューチェックリスト

| 質問 | 回答 |
|------|------|
| Q1: クライアントは修復代数を計算するか？ | **いいえ。** クライアントは無知な再送信エージェントである。すべての疑い導出、信頼度スコア、コスト閾値、収縮ロジックはサーバー側専用である。 |
| Q2: 収縮前に修復コスト閾値が明示的に評価されるか？ | **はい。** `htp_repair_worthwhile` は `retry_cost_estimate`（バイト数 × RTT）と `repair_cost_estimate`（サーバー分析オーバーヘッド）を比較する。`疑い数 < 3` または再試行の方が安価な場合 false を返し、直接再試行にフォールバックする。 |
| Q3: 部分グループがゼロパディングされるか？ | **いいえ。** 完全な 6 スロットグループ（`chunk_count / 6`）のみが検証される。不完全な最終グループは完全に除外される。 |
| Q4: レスポンスにバイナリフラグではなく疑いスコアが含まれるか？ | **はい。** すべての `needs_retry` レスポンスは `suspicion_scores` を `{chunk_index, score}` オブジェクトとして含む。 |
| Q5: 収縮が元のグループトポロジーを保持するか？ | **はい。** `htp_contract_groups` は各元の完全グループを単一の高次頂点として扱い、疑いチャンクをグループ間で再シャッフルしない。 |
| Q6: 成功した再送信時に再試行対象が消去されるか？ | **はい。** `handler_file_upload` は再試行再送信を受け入れた後、`htp_retry_targets` から当該チャンクを削除する。 |
| Q7: サーバーは順序が逆になったスカラーを swap で直接修復するか？ | **はい。** `htp_try_swap_repair` は 6 スロットグループの 720 通りの全順列を力任せに探索し、不変式を復元する順列があれば `htp.bin` レコードを即座に並び替え、そのグループを再送信なしで通過させる。 |
