# fly.board

![fly.board logo](img/logo.png)

> Один из немногих простых движков блогов, у которых потребление памяти почти не растёт с ростом числа соединений: **~82 МБ RSS** в простое (4 воркера; на реальном production-сервере с одним воркером поддерживает **68–120 МБ**), и всё ещё **~146 МБ** под нагрузкой C10k, C100k и даже C1m.  
> Лёгкий гибрид форума и блога на основе C-фреймворка CWIST с поддержкой HTTPS/3, Argon2id, PQC-подписей и NATS.

## Возможности

- **Эффективное использование памяти и масштабируемость по соединениям** — реализация на C со стеком и кучей. **~82 МБ RSS** в простое; RSS остаётся около **~146 МБ** от C10k до C1m одновременных соединений.
- **Современный транспорт** — TLS 1.3 + HTTP/3 (QUIC) по умолчанию. Опциональный ECH (Encrypted Client Hello).
- **Безопасная аутентификация** — клиентский прехеш SHA-512 + серверный **Argon2id** (KDF из OpenSSL 3). JWT-куки для сессий.
- **Гибрид форум/блог** — Markdown-посты на основе slug + несколько досок + вложенные комментарии.
- **Предпросмотр в реальном времени** — мгновенный рендеринг предпросмотра на стороне сервера из Markdown-редактора.
- **PQC-подписи** — прикрепление и проверка постквантовых криптографических подписей к постам.
- **Файловое хранилище** — ≤1 МБ в SQLite, файлы больше — на томе. Автовстраивание изображений/видео/аудио.
- **Интеграция NATS** — распределённый шлюз сообщений через переменную окружения `NATS_URL`.
- **Тёмная тема** — переключение темы на основе Cookie с динамическими CSS-переменными.

## Сборка

```sh
make
./keygen.sh
```

Зависимости:
- [CWIST](https://github.com/religiya-serdtsa/cwist) — TLS 1.3 / HTTP/3 (QUIC) обрабатывается встроенным BoringSSL внутри CWIST; дополнительная установка не требуется.
- OpenSSL 3.x (Argon2id KDF)
- ngtcp2 / nghttp3 (HTTP/3)
- cJSON, SQLite3

`Makefile` клонирует и собирает `third_party/md4c` в виде статической библиотеки.

## Запуск

```sh
./fly_board
```

Порт по умолчанию берётся из значения `port` в файле `blog.settings` (по умолчанию 9443).

```text
https://localhost:9443
```

HTTP/3 слушает на том же порту по UDP.

### Включение ECH (опционально)

```sh
BLOG_ECH_KEY=ech/server.ech ./fly_board
# или
BLOG_ECH_DIR=ech ./fly_board
```

Если сборка OpenSSL не поддерживает ECH, в лог будет записано предупреждение, а сервер продолжит работу с обычным HTTPS/3.

### Интеграция NATS (опционально)

```sh
NATS_URL=nats://localhost:4222 ./fly_board
```

## Ключевые возможности

| Возможность | Путь | Описание |
|---------|------|-------------|
| Главная | `/` | Список последних постов |
| Доски | `/boards` | Управление несколькими досками (только для администраторов) |
| Пост | `/post/:slug` | рендеринг md4c Markdown + комментарии + вложения |
| Вход/Регистрация | `/login`, `/register` | Argon2id + JWT-куки |
| Профиль | `/profile` | Никнейм, биография, аватар, дата регистрации |
| Настройки аккаунта | `/account/settings` | Редактирование профиля |
| Смена пароля | `/account/password` | Проверка текущего пароля, повторный хеш Argon2id |
| Админка | `/admin/users` | Изменение ролей пользователей, удаление пользователей |
| Файловое хранилище | `/files` | Загрузка/скачивание/удаление |

## Конфигурация

- `blog.settings` — заголовок блога, подзаголовок, подвал, порт и лимиты загрузки
- `admin.settings` — учётная запись администратора (2 строки: `username`\n`password`)

## База данных

SQLite3 (`data/blog.db`). Схема автоматически мигрируется при запуске приложения.

```
users       — аккаунты, хеши Argon2id, роли, профили
boards      — название/slug/описание доски/admin_only
posts       — тело Markdown, PQC-подпись, краткое содержание
files       — путь/размер/MIME вложения
comments    — вложенные комментарии (target_type, parent_id)
board_permissions — права доступа к приватным доскам
```

## Архитектура

```
CWIST (HTTP/3, TLS 1.3)
  ├── src/auth/     — Argon2id, JWT, сессии
  ├── src/db/       — SQLite3 CRUD
  ├── src/handlers/ — маршрутизация/бизнес-логика
  ├── src/render/   — cwist_html_element SSR + md4c
  ├── src/crypto/   — PQC подпись/проверка
  └── src/nats/     — Pub/Sub сообщений
```

## Бенчмарк масштабируемости

### Что измеряет этот бенчмарк

Эти тесты используют `h2load` **с опцией `-r`** (ограничение скорости). Они намеренно **не** являются тестами максимальной пропускной способности. Вместо этого они измеряют, может ли сервер **удерживать огромное количество одновременных HTTP/2-соединений**, обрабатывая контролируемую скорость запросов на процесс.

Поскольку нагрузка ограничена по скорости:

- Отображаемый **RPS отражает настроенную скорость запросов**, а не абсолютный потолок пропускной способности сервера.
- Главный метрика — **стабильность resident-set-size (RSS)** по мере роста числа соединений от 10 000 до 1 000 000.

Число воркеров масштабируется вместе с нагрузкой, чтобы каждый тест оставался реалистичным: **4 воркера** для C10k, **12 воркеров** для C100k и **24 воркера** для C1m. Это также объясняет различия в показателях использования CPU между тремя запусками.

### Использование памяти

| Состояние | RSS | Δ от предыдущего | Примечания |
|-------|-----|-----------------|-------|
| Простой | **~82 МБ** (83,708 КБ) | — | 4 workers, no connections |
| C10k | **~146 МБ** (145,928 КБ) | +62.22 MB | 10,000 concurrent connections |
| C100k | **~146 МБ** (146,076 КБ) | +148 KB | 100,000 concurrent connections |
| C1m | **~146 МБ** (146,420 КБ) | +344 KB | 1,000,000 concurrent connections |

Общий прирост RSS **от C10k до C1m составляет всего ~492 КБ** — по сути, шум. Это самый важный результат бенчмарка.

Значения RSS — это **Maximum resident set size (kbytes)**, сообщаемые `/usr/bin/time -v` для процесса сервера.

### Стоимость памяти

| Переход | Δ RSS | Δ соединений | Приблизительная стоимость на одно дополнительное соединение |
|---|---|---|---|
| Idle → C10k | +62.22 МБ | 10 000 | ~6.4 КБ / соединение |
| C10k → C1m | +492 КБ | 990 000 | ~0.5 байта / дополнительное соединение |

Начальный скачок от Idle до C10k — это авансовая оплата за состояние TLS, буферы соединений и накладные расходы воркеров. После этого добавление ещё 990 000 соединений стоит менее полубайта RSS на каждое — стоимость памяти на соединение практически не растёт.

**Ключевые выводы**

- **Память держится и на C1m**: даже когда тестовое железо не могло полностью обслужить все 1 000 000 соединений (36.14% успеха), использование памяти осталось практически неизменным — сервер не вышел из-под контроля.
- **Безопасность данных**: SQLite безопасно сохранила все данные при SIGINT (10,600 FS outputs at C10k).

### Бенчмарк пропускной способности

Бенчмарк выше измеряет **connection scalability**, а не абсолютную **request throughput**. Чтобы измерить сырой потолок пропускной способности сервера, был запущен неограниченный тест с помощью `h2load` (без ограничения скорости `-r`) поверх HTTP/2.

| Item | Value |
|------|-------|
| Command | `h2load -c512 -n100000 https://127.0.0.1:8888/` |
| Workers | 12 |
| Concurrent connections | 512 |
| Total requests | 100,000 |
| Succeeded | 100,000 |
| Failed / Errored / Timeout | 0 |
| Duration | 13.95 s |
| Mean RPS | **7167.28** |
| Mean throughput | **290.51 MB/s** |
| Задержка запроса (h2load `time for request`) | min 183 µs, mean 30.69 ms, max 209.00 ms, sd 11.18 ms |
| Приблизительная перцентильная задержка* | p50 ~30.7 ms, p95 ~49.1 ms, p99 ~56.7 ms |

\* Перцентили оценены по сообщённым среднему и стандартному отклонению; h2load по умолчанию выводит только min/max/mean/sd. Для точной гистограммы перцентилей запустите с опцией `--latency-collect`.

#### Сравнение с HTTP/1.1 через `wrk`

Для сравнения, тот же endpoint был протестирован с помощью `wrk` поверх HTTP/1.1. Поскольку протокол и инструмент разные, приведённые ниже цифры **нельзя напрямую сравнивать** с результатами HTTP/2 h2load выше.

| Item | Value |
|------|-------|
| Command | `wrk -t12 -c512 -d60s https://127.0.0.1:8888/` |
| Duration | 60 s |
| Requests/sec | **1282.49** |
| Transfer/sec | 52.29 MB |
| Latency | Avg 138.61 ms, Stdev 39.26 ms, Max 311.70 ms |

Эти цифры показывают абсолютный потолок пропускной способности движка под сфокусированной неограниченной нагрузкой. Они отделены от тестов масштабируемости по соединениям выше.
