# TASFA 檔案傳輸協定分析

## TASFA 是什麼

TASFA 是 fly.board 使用的 **基於 HTTP/XHR 的檔案上傳/下載協定**。它不僅僅是檔案管道，而是一個整合了分塊加密、完整性校驗、不穩定網路應對與 DoS 防護的協定層。

## 核心構成要素

### 1. 分塊 AES-256-GCM 串流加密

- 每個上傳工作階段生成唯一的 `stream_key` 與 `iv_seed`。
- 每個資料區塊（chunk）使用 `AES-256-GCM` 加密，並以 `upload_id:chunk_index` 作為 AAD（附加認證資料）。
- 標籤（tag）驗證失敗時，該資料區塊會被立即丟棄。
- 伺服器收到密文後解密，並將明文直接寫入預先分配的暫存檔案偏移量 `chunk_index * chunk_size` 處。
- 此加密確保了**工作階段間獨立性**：某一工作階段的金鑰洩露不會影響其他工作階段。但這並非替代 TLS，而是工作階段層級的附加完整性校驗層。

### 2. 6 槽 HTP（Hexagonal Tortoise Problem）本地修復 Lattice

- HTP **不是傳輸協定**，而是**本地修復提示（hint）**。
- 資料區塊按每 6 個一組進行分組。
- 每個資料區塊的 SHA-512 前 8 位元組被解釋為 64 位元整數 `H`，計算 `raw_scalar = H % modulus_M`。
- 完整組需滿足三條直線不變式（`L1 == L2 == L3`）；若不滿足，僅調整 `v3` 與 `v5` 使其以最小增量達到平衡。
- 上傳完成時，伺服器逐組驗證不變式。不匹配則傳回 `400 Bad Request`。
- 這提供了一種輕量級早期檢測手段，可在**資料區塊層級的替換或損壞**尚未累積到整檔案雜湊失效前即發現異常。

### 3. 檔案級順序佇列（瀏覽器連線穩定性）

- **同一時間僅上傳一個檔案**。
- 多檔案選取時，檔案進入 `FileUploadQueue` 排隊，只有目前檔案成功或失敗後，下一個檔案才開始傳輸。
- 此舉尊重瀏覽器對同一來源（origin）6 個並行連線的限制，防止連線池耗盡導致的死結。

### 4. 二進位元點陣圖用於 DoS 緩解與狀態追蹤

- 上傳進度以**稠密二進位元點陣圖**形式儲存在 `state.bin` 中（每個資料區塊 1 位元組，`'0'`/`'1'`）。
- 已標記為 `'1'` 的資料區塊索引若再次收到重傳，伺服器直接拒絕。由此阻止透過重複傳送同一區塊來消耗磁碟 I/O 的 DoS 攻擊。
- 使用 `pwrite(..., 1, chunk_index)` 實作 O(1) 原子寫入，熱路徑上**不存在 JSON 解析**。
- 下載側同樣維護點陣圖，已獲取的資料區塊跳過，並快取至以 `baseUrl + ':' + chunk_index` 為鍵的 `IndexedDB` 中。重試風暴（retry storm）期間不會重複請求相同資料。

### 5. 適應型步速（Adaptive Pacing）

- 網路狀況惡化時，TASFA 並非簡單降低並行度，而是根據鏈路品質在資料區塊排程之間注入 `setTimeout` 延遲（pacing）。

| 鏈路評分 | 上傳 pacing | 下載 pacing | Coalesce |
|-----------|---------------|-----------------|----------|
| >= 85     | 0 ms          | 0 ms            | 64       |
| 65 – 84   | 15 ms         | 0 ms            | 48       |
| 45 – 64   | 15 ms         | 10 ms           | 32       |
| < 45      | 35 ms         | 30 ms           | 16       |

- 防止高丟包鏈路下 TCP 視窗崩潰，避免緩衝區膨脹（buffer-bloat）導致的傳輸停滯。
- 鏈路評分低於 45 時，逾時時間延長（最高 12 秒），重試退避係數從 1.6 降至 1.3。

### 6. 工作階段強化

- 上傳 ID 與 Token 分別為 16 位元組與 24 位元組隨機十六進位字串。
- 使用 `flock` 防止並行請求間的點陣圖更新競爭。
- 下載工作階段受 TTL 限制（`TASFA_DOWNLOAD_TTL = 86400`），過期後自動丟棄。

## RPS 犧牲是否值得

> **注意**：以下 RPS 權衡**僅適用於檔案傳輸端點**（`/file/upload`、`/file/download` 等）。基準測試中測得的首頁、看板列表等頁面 RPS 不受 TASFA 影響。

與簡單的靜態檔案服務相比，TASFA 顯然會帶來 **RPS（每秒請求數）損失**。然而，這種損失是為了換取以下價值而有意做出的權衡：

| 犧牲的內容 | 獲得的內容 |
|------------------|---------------|
| 熱路徑上的額外加解密開銷 | **分塊完整性**：標籤不匹配立即被偵測到，損壞的資料區塊不會落盤 |
| HTP 純量計算與分組驗證消耗的 CPU | **早期損壞偵測**：整檔案雜湊只能在傳輸結束後才能發現的錯誤，可在分組層級提前捕獲 |
| 位點陣圖檔案 I/O 與 `flock` 工作階段鎖 | **DoS 防護**：阻止重複傳送同一區塊的濫用，以及截斷檔案攻擊（truncated-file attack） |
| 順序佇列導致無法同時上傳多個檔案 | **瀏覽器連線穩定性**：在單一 origin 的 6 連線限制內實現無死結的穩定傳輸 |
| pacing 帶來的較低資料區塊傳送速率 | **不穩定鏈路恢復力**：在高丟包/高延遲環境下仍能持續傳輸，避免 TCP 視窗崩潰 |
| IndexedDB 快取與工作階段狀態佔用的記憶體與磁碟 | **重試效率化**：網路斷線後重新連線時，已接收的資料區塊不會被重複請求 |

### 結論

TASFA 並不追求「最快的檔案傳輸」。相反，它的設計目標是：**即使在不穩定網路和惡意用戶端環境下，檔案也能完整、無損地到達目的地**。

- **上傳視角**：工作階段級加密資料流 + HTP 提示 + 位點陣圖 DoS 防護 + 順序佇列 = 穩定且無損的上傳。
- **下載視角**：基於交握的工作階段 + 位點陣圖追蹤 + IndexedDB 快取 + 適應型步速 = 不間斷的下載。

對於純部落格而言 TASFA 可能過於複雜；但對於**以附件為核心的社群論壇**，這種 RPS 損失具備明確的技術合理性，值得承擔。
