# TASFA

TASFA — это протокол передачи файлов, используемый в данном проекте для загрузки и скачивания.

Он построен поверх обычного HTTP/XHR и добавляет шифрование по частям, последовательную очередь на уровне файла, а также **серверную решётку восстановления HTP (Hexagonal Tortoise Problem) с 6 слотами** для контроля целостности.

## Маршруты

Загрузка:
- `POST /file/upload/init`
- `POST /file/upload/status`
- `POST /file/upload/renegotiate`
- `POST /file/upload`
- `POST /file/upload/complete`
- `POST /file/upload/cancel`

Скачивание:
- `GET /file/download/:id/handshake`
- `GET /file/download/:id/chunk/:chunk_index`
- `GET /assets/tasfa/img/:filename/handshake`
- `GET /assets/tasfa/img/:filename/chunk/:chunk_index`
- `GET /assets/tasfa/uploads/:filename/handshake`
- `GET /assets/tasfa/uploads/:filename/chunk/:chunk_index`

## Протокол Загрузки

Браузер сначала согласовывает сессию загрузки, затем отправляет **чанки** (по умолчанию `24 MiB`, на мобильных `12 MiB`) с заголовками TASFA:

- `X-TASFA-Upload-ID`
- `X-TASFA-Upload-Token`
- `X-TASFA-Chunk-Index`
- `X-TASFA-Hash-Tag`
- `X-TASFA-Magic-Scalar`

Сервер записывает каждый чанк непосредственно в предварительно выделенный временный файл по смещению `chunk_index * chunk_size`. Транспортных блоков больше нет.

Если обычный чанк многократно завершается с ошибкой, браузер отправляет этот неудавшийся чанк через запрос отката AES-256-GCM с `X-TASFA-Stream-Mode: aes-256-gcm`. Сервер принимает шифротекст вместе с тегом аутентификации GCM, и чанки отката остаются внутри адаптивного параллельного окна. Запрос отката по-прежнему содержит те же заголовки хеш-тега HTP и сбалансированного скаляра.

### Остаток (последний частичный чанк)

Если размер файла не кратен размеру чанка, последний чанк является **остатком**. Он отправляется как единый blob с точным диапазоном байтов; сервер записывает его по правильному смещению. Заполнение или разделение не выполняется.

### Ответ инициализации сессии

Конечная точка `init` возвращает, среди прочего:

- `chunk_size` — согласованный размер чанка
- `max_parallel_chunks` — сколько чанков клиент может загружать параллельно

## Серверная Решётка Восстановления HTP

HTP **не является транспортным протоколом и не является криптографическим доказательством**. Это серверный движок подозрения в отношении чанков, который ранжирует чанки по вероятности повреждения, чтобы клиент повторно передавал только высоковероятных подозреваемых вместо всего файла.

### Группировка чанков

Чанки группируются в последовательные вершины из 6 элементов:

```
Группа g: [ v0 , v1 , v2 , v3 , v4 , v5 ]
            chunk g*6+0  ...  g*6+5
```

Последняя группа может быть неполной; **неполные группы никогда не заполняются нулями и полностью исключаются из валидации HTP**. Заполнение нулями внедряло бы синтетическую топологию и запрещено.

### Исходный скаляр

Для каждого чанка клиент вычисляет SHA-512 открытого текста чанка, берёт первые 8 байтов как беззнаковое 64-битное целое число big-endian `H` и выводит:

```
raw_scalar = H % modulus_M
```

### Инвариант магической линии

Для полной группы определяются три линии:

```
L1 = v0 + v1 + v2   (mod M)
L2 = v2 + v3 + v4   (mod M)
L3 = v4 + v5 + v0   (mod M)
```

Инвариант требует `L1 == L2 == L3`. Если исходные скаляры не удовлетворяют этому, клиент **балансирует**, корректируя только `v3` и `v5`:

```
delta2 = (L1 - L2) mod M
delta3 = (L1 - L3) mod M

v3_balanced = (v3_raw + delta2) mod M
v5_balanced = (v5_raw + delta3) mod M
```

Все остальные вершины сохраняют свой исходный скаляр. Клиент отправляет оба:\n\n- `X-TASFA-Raw-Scalar` — немодифицированный `raw_scalar`\n- `X-TASFA-Magic-Scalar` — сбалансированное значение (`v3_balanced` или `v5_balanced`, в остальных случаях совпадает с raw)\n\nСервер хранит оба скаляра отдельно в `htp.bin`, чтобы анализ мог ссылаться на исходную топологию независимо от искусственных ограничений балансировки.

### Почему только v3 и v5?

Шестиугольная решётка имеет две степени свободы. Фиксируя `v0,v1,v2,v4` и корректируя `v3,v5`, можно единственным образом удовлетворить трём уравнениям линий, сохраняя минимальную дельту и локальность внутри группы.

## Серверное Восстановление HTP

**Клиент — это тупой агент повторной передачи.** Он не вычисляет алгебру восстановления, не оценивает пороги стоимости и не выводит ранжирование подозреваемых. Всё это находится исключительно на сервере.

### Поток валидации сервера

Во время `POST /file/upload/complete` сервер:

1. Загружает все `raw_scalars` и `magic_scalars` для каждого чанка из `htp.bin`.
2. Проверяет только **полные группы из 6 слотов** (частичные группы пропускаются).
3. Для каждой неудавшейся группы вычисляет **оценки подозрения** для каждого слота, анализируя участие в уравнениях линий.

### Оценка достоверности подозрения (по группам)

Для неудавшейся группы сервер оценивает каждый слот по отношению к трём уравнениям линий:

| Слот | Уравнения |
|------|-----------|
| v0   | L1, L3    |
| v1   | L1        |
| v2   | L1, L2    |
| v3   | L2        |
| v4   | L2, L3    |
| v5   | L3        |

Подозрительный балл каждого слота является детерминированным и выводится исключительно из топологии:

```
score = in_fail / total_fail
```

где `in_fail` — количество неудачных линейных уравнений, в которых участвует слот, а `total_fail` — общее количество неудачных уравнений в данной группе. Произвольные константы доверия не используются.

Если слот появляется только в пройденных уравнениях, он **исключается** из списка подозреваемых.

Оценки агрегируются по всем неудавшимся группам; если чанк появляется в нескольких группах, сохраняется максимальная оценка.

### Порог стоимости восстановления

Перед запросом любого восстановления сервер оценивает, дешевле ли сжатие, чем прямая повторная передача:

```
repair_worthwhile(подозреваемые, всего, размер_чанка, rtt_ms):
    if подозреваемые < 3                → false  (слишком мало для топологии)
    retry_cost  = подозреваемые * размер_чанка * rtt_factor(rtt_ms)
    repair_cost = метаданные_байты + cpu_сервера + доп_rtt_стоимость
    return retry_cost > repair_cost
```

Абстрактная модель затрат сравнивает байты, которые клиент должен повторно передать, с накладными расходами на анализ на стороне сервера. Большие чанки или высокая задержка делают сжатие более привлекательным, а множество мелких подозреваемых — прямой повтор дешевле. Конкретные числа являются деталями реализации сервера, а не константами протокола.

Если порог отклоняет восстановление, сервер возвращает `needs_retry` со **всеми** подозреваемыми чанками в качестве целей повторной передачи. Клиент повторно передаёт их через обычную конечную точку загрузки.

### Серверное рекурсивное сжатие

Если восстановление целесообразно, сервер выполняет **сжатие на уровне групп**: каждая оригинальная полная 6-слотовая группа сворачивается в один скаляр (сумма её балансированных скаляров по модулю `M`). Эти групповые агрегаты становятся вершинами решётки HTP более высокого уровня. Последовательные группы по 6 таких агрегатов формируют супер-группы уровня 1, и те же линейные инварианты переоцениваются:

- Если супер-группа уровня 1 проходит, подозреваемые из её базовых групп уровня 0 очищаются (паттерн отказа согласован на уровне групп).
- Если супер-группа уровня 1 не проходит, подозреваемые из её базовых групп уровня 0 сохраняются.
- Если сжатие сужает множество подозреваемых (меньше чанков), сервер сохраняет суженные цели и возвращает `needs_retry` с уменьшенным списком.
- Если сжатие не сужает множество, сервер откатывается к прямой повторной передаче исходных подозреваемых.
- Уровень сжатия инкрементируется в метаданных сессии, чтобы клиент мог сообщать диагностику.

Клиент никогда не видит группы сжатия и не вычисляет их. Он получает только `retry_targets`.

### Принятие повторной передачи

Когда клиент повторно передаёт чанк, уже помеченный как полученный, обычная конечная точка загрузки **принимает повторную передачу только если индекс этого чанка в данный момент находится в списке `retry_targets` сервера**. После сохранения повторно переданного чанка сервер удаляет его из `retry_targets`.

### Видимый протоколу ответ восстановления

При сбое HTP, когда сервер принимает решение о восстановлении или повторной передаче, конечная точка `complete` возвращает `409` с:

- `htp_status`: `"needs_retry"`
- `retry_targets`: массив индексов чанков для повторной передачи (упорядочен по убыванию оценки подозрения)
- `suspicion_scores`: массив объектов `{chunk_index, score}`
- `contraction_level`: сколько проходов серверного сжатия было применено
- `retry_reason`: читаемое человеком объяснение (например, `"htp group inconsistency detected"`)

Если порог стоимости говорит, что прямая повторная передача дешевле, `retry_targets` содержит полный список подозреваемых, а `contraction_level` остаётся `0`.

Если сервер успешно сужает подозреваемых через сжатие, `retry_targets` содержит суженный список, а `contraction_level` увеличивается.

После того как все подозреваемые чанки повторно переданы и успешно проверены, следующий вызов `complete` переходит к финализации SHA-256.

## Последовательная Очередь на Уровне Файла

**Одновременно загружается только один файл.** При выборе нескольких файлов:

1. Каждый файл получает свой собственный ресурс, карточку предпросмотра и сессию HTP.
2. Файлы ставятся в очередь `FileUploadQueue`.
3. Когда активный файл завершается (успех или неудача), очередь автоматически переходит к следующему файлу.
4. Кнопка пакетной загрузки «Загрузить файлы в очереди» всегда активна; при нажатии она ставит в очередь все ожидающие файлы и запускает насос.

Это предотвращает исчерпание пула соединений браузера и поддерживает надёжное обнаружение зависаний.

## Настройки Времени Выполнения

- размер чанка загрузки: `24 MiB` на десктопе, `12 MiB` на мобильных
- параллелизм загрузки браузера по умолчанию: `16`
- максимальный параллелизм загрузки браузера: `max_upload_parallel_chunks` в `blog.settings`
- максимальное количество параллельных сессий загрузки: `max_total_parallel_uploads` в `blog.settings`
- максимальный размер загрузки: `max_upload_size` в `blog.settings`
- максимальное количество сессий скачивания браузера: определяется сервером
- тайм-аут xhr загрузки: не менее `180 с`
- тайм-аут fetch сессии загрузки: `30 с`

Ограничение HTTP-соединений браузера на origin естественным образом соблюдается пулом воркеров.

### Адаптация Клиента

Клиент загрузки измеряет время завершения чанка, повторные попытки, таймауты и подсказки API информации о сети, когда они доступны. Он отправляет эти данные на `/file/upload/init` и `/file/upload/renegotiate`; сервер отвечает текущим параллельным окном и максимальным окном. Чистые завершения быстро увеличивают активное окно до согласованного максимума, в то время как временные сбои больше не обрушиваются ниже небольшого порога параллелизма. Откат AES-GCM также выполняется внутри адаптивного окна. Watchdog прерывает и возобновляет зависшую загрузку с серверного битмапа, вместо того чтобы оставлять передачу застрявшей.

Загрузка использует тот же высокопроизводительный уклон: рукопожатие несёт предпочтительную подсказку размера чанка клиента, и активные загрузки увеличивают `span` и параллелизм после успешных групп чанков. Короткие чтения, таймауты и сетевые ошибки ставятся обратно в очередь по индексу чанка; неудавшаяся группа не приводит к провалу всей загрузки, пока тот же чанк не исчерпает высокий бюджет повторных попыток.

### Предсказание RTT Хвоста (Экстраполяция Лагранжа)

Для больших файлов (сессий с большим количеством чанков) сервер накапливает образцы RTT по чанкам, сообщаемые клиентом, максимум до 8. Как только собрано 3 или более образцов, полиномиальная экстраполяция Лагранжа оценивает RTT на индексе последнего чанка, и количество оставшихся чанков умножается для получения оценочного оставшегося времени.

- Загрузка: после завершения каждого чанка клиент может прикрепить заголовок `X-TASFA-Chunk-RTT` (миллисекунды) к следующему запросу чанка.
- Скачивание: клиент может отдельно сообщить RTT предыдущего чанка через параметр запроса `chunk_rtt_ms`.

Сервер добавляет `predicted_remaining_ms` в JSON-ответ состояния, а ответ на запрос чанка скачивания содержит заголовок `X-TASFA-Predicted-Remaining-Ms`. Математика намеренно остаётся лёгкой: ограничение в 8 образцов, простой O(n²) Лагранж, и предсказания ограничиваются потолком в 30 секунд.

### Предиктивная Защита с Ускорением Винна

TASFA не предсказывает байты открытого текста и не доверяет предсказанному содержимому чанков. Предсказание ограничено последовательностью качества передачи, наблюдаемой клиентом: время завершения, пропускная способность, повторные попытки, таймауты и короткие чтения. Клиент применяет epsilon-преобразование Винна в форме Aitken `Delta^2` к последним образцам качества:

```
S_hat = S0 - ((S1 - S0)^2 / (S2 - 2*S1 + S0))
```

`S_hat` ограничивается диапазоном `[0, 1]` и используется только как защитный сигнал для следующего чанка. Высокое предсказанное качество сохраняет обычный путь и позволяет адаптивному окну расти. Средне-низкое предсказанное качество помечает следующий чанк как `guarded`, уменьшая активное параллельное окно на один перед отправкой. Очень низкое предсказанное качество помечает следующий чанк загрузки для параллельного отката AES-GCM, или уменьшает `span` и параллелизм загрузки перед выдачей следующей группы диапазонов. Это опережающая защита от перегрузки и сбоев, а не криптографический оракул.

## Модель Хранения

Каждая сессия загрузки предварительно выделяет один временный файл:

- временный файл: `data/tasfa/uploads/<upload_id>/upload.bin.part`
- метаданные: `data/tasfa/uploads/<upload_id>/meta.json`
- быстрая бинарная мета: `data/tasfa/uploads/<upload_id>/meta.bin`
- состояние: `data/tasfa/uploads/<upload_id>/state.json`, `data/tasfa/uploads/<upload_id>/state.bin`
- HTP уровень 0: `data/tasfa/uploads/<upload_id>/htp.bin`

Сервер больше не поддерживает `blocks.bin` или `chunk_counts.bin`. Завершение чанка — это одна запись битовой карты.

Метаданные сессии также хранят массивы по вершинам:

- `hash_tags` — массив шестнадцатеричных строк SHA-512, по одной на чанк
- `raw_scalars` — массив raw-скаляров (немодифицированный дайджест SHA-512 mod M), по одному на чанк\n- `magic_scalars` — массив сбалансированных скаляров, по одному на чанк
- `htp_retry_targets` — текущий список целей повторной передачи, выданный сервером
- `htp_suspicion_scores` — текущее ранжирование подозрения
- `htp_contraction_level` — количество применённых проходов серверного сжатия

Они обновляются при каждой загрузке чанка и проверяются при завершении.

## PIN Удаления

Завершённые загрузки получают одноразовый PIN удаления. Чистый PIN возвращается один раз; хранится только его хеш.

## Протокол Скачивания

1. Клиент запрашивает рукопожатие через `GET /.../handshake`.
2. Сервер возвращает `session_id`, `session_token`, `chunk_size`, `chunk_count`, подсказки параллелизма и сессионные ключи шифрования (`stream_key_hex`, `stream_iv_seed_hex`).
3. Клиент получает группы чанков с адаптивным `span=...`. Все чанки зашифрованы с помощью **AES-256-GCM**.
4. Клиент расшифровывает чанки в браузере, используя **Web Crypto API** и ключи сессии.
5. Браузер собирает ответ в один непрерывный буфер.

## Интеграция обработки медиа

Медиафайлы, создаваемые сервером (миниатюры, аудио-превью), являются первоклассными ресурсами TASFA:
- **Предварительный расчет метаданных HTP**: Когда сервер создает миниатюру или превью, он немедленно вычисляет скаляры HTP и теги SHA-256 для их чанков и сохраняет их в `data/tasfa/media_htp`.
- **Надежная передача медиа**: Медиафайлы обслуживаются через маршруты `/assets/tasfa/...`, которые поддерживают полный протокол TASFA, включая проверку целостности на уровне чанков с использованием предварительно рассчитанных метаданных HTP.
- **Управление параллелизмом**: Генерация медиа (ffmpeg) ограничена 4 одновременными процессами для защиты ресурсов сервера.

## Защита от DoS через Битовую Карту

И состояние загрузки, и состояние скачивания отслеживаются с помощью **плотной бинарной битовой карты** (1 байт на чанк, `'0'` / `'1'`).

### Сторона загрузки

- Сервер отклоняет любой чанк, индекс которого уже помечен `'1'` в `state.bin`, **если только этот чанк не находится явно в списке целей повторной передачи сервера**. Злоумышленник не может воспроизводить произвольные чанки для расходования дискового I/O.
- `state.bin` обновляется через `pwrite(..., 1, chunk_index)` — атомарная запись за O(1). На критическом пути отсутствует JSON-парсинг.
- Обработчик `complete` повторно открывает битовую карту, считает установленные биты и отказывается финализировать до тех пор, пока `received_chunks == chunk_count`. Это предотвращает атаки с усечёнными файлами.

### Укрепление сессий

- ID загрузки и токены — это случайные шестнадцатеричные строки длиной 16 и 24 байта.
- Блокировки сессий (`flock`) предотвращают состязательные обновления битовой карты из параллельных запросов.

## Контрольный Список Самопроверки

| Вопрос | Ответ |
|--------|-------|
| Q1: Вычисляет ли клиент какую-либо алгебру восстановления? | **Нет.** Клиент — это тупой агент повторной передачи. Вся деривация подозреваемых, оценка достоверности, пороги стоимости и логика сжатия являются исключительно серверными. |
| Q2: Является ли оценка порога стоимости восстановления явной перед сжатием? | **Да.** `htp_repair_worthwhile` сравнивает `retry_cost_estimate` (байты × RTT) с `repair_cost_estimate` (серверные накладные расходы анализа). Он отклоняет восстановление, когда `suspect_count < 3` или прямая повторная передача дешевле, возвращаясь к прямой повторной передаче. |
| Q3: Заполняются ли частичные группы нулями? | **Нет.** Проверяются только полные группы из 6 слотов (`chunk_count / 6`). Неполная последняя группа полностью исключается. |
| Q4: Содержит ли ответ оценки подозрения, а не только бинарные флаги? | **Да.** Каждый ответ `needs_retry` включает `suspicion_scores` как объекты `{chunk_index, score}`. |
| Q5: Сохраняет ли сжатие исходную топологию групп? | **Да.** `htp_contract_groups` обрабатывает каждую оригинальную полную группу как единую вершину высшего уровня; подозреваемые чанки никогда не перетасовываются между группами. |
| Q6: Очищаются ли цели повторной передачи при успешной повторной передаче? | **Да.** `handler_file_upload` удаляет чанк из `htp_retry_targets` после принятия повторной передачи. |
