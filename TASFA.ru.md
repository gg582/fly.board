# Анализ протокола передачи файлов TASFA

## Что такое TASFA

TASFA — это **протокол загрузки/скачивания файлов на базе HTTP/XHR**, используемый в fly.board. Это не просто канал для файлов, а протокольный слой, объединяющий шифрование по частям, проверку целостности, работу в нестабильных сетях и защиту от DoS.

## Основные компоненты

### 1. Потоковое шифрование AES-256-GCM по чанкам

- Для каждой сессии загрузки генерируются уникальные `stream_key` и `iv_seed`.
- Каждый чанк шифруется с помощью `AES-256-GCM`, в качестве AAD используется `upload_id:chunk_index`.
- При неудаче проверки тега чанк немедленно отбрасывается.
- Сервер получает шифртекст, расшифровывает его и записывает открытый текст напрямую во временный файл по смещению `chunk_index * chunk_size`.
- Это шифрование обеспечивает **изоляцию сессий**: компрометация ключа одной сессии не влияет на другие. Однако это не замена TLS, а дополнительный уровень проверки целостности на уровне сессии.

### 2. 6-слотовая решётка HTP (Hexagonal Tortoise Problem) для локального восстановления

- HTP — **это не транспортный протокол**, а **подсказка для локального восстановления**.
- Чанки группируются по 6 элементов.
- Первые 8 байт SHA-512 каждого чанка интерпретируются как 64-битное целое `H`, вычисляется `raw_scalar = H % modulus_M`.
- Для полной группы должно выполняться условие инвариантности трёх линий (`L1 == L2 == L3`); если нет, корректируются только `v3` и `v5` для достижения баланса с минимальным изменением.
- По завершении загрузки сервер проверяет инварианты для каждой группы. Несоответствие возвращает `400 Bad Request`.
- Это обеспечивает лёгкое раннее обнаружение **замены или повреждения на уровне чанка**, которое хэш всего файла не может выявить до самого конца.

### 3. Последовательная очередь на уровне файла (стабильность соединений браузера)

- **Одновременно загружается только один файл**.
- При выборе нескольких файлов они ставятся в очередь `FileUploadQueue`. Следующий файл начинается только после успеха или неудачи текущего.
- Это соблюдает ограничение браузера в 6 параллельных соединений на origin и предотвращает взаимоблокировки из-за исчерпания пула соединений.

### 4. Двоичная битовая карта для смягчения DoS и отслеживания состояния

- Прогресс загрузки хранится в `state.bin` как **плотная двоичная битовая карта** (1 байт на чанк, `'0'`/`'1'`).
- Сервер отклоняет любой чанк, индекс которого уже помечен `'1'`. Это блокирует DoS-атаки, повторно отправляющие один и тот же чанк для сжигания дискового I/O.
- Обновления выполняются через `pwrite(..., 1, chunk_index)` — атомарная запись за O(1) **без парсинга JSON на горячем пути**.
- На стороне скачивания также поддерживается битовая карта: уже полученные чанки пропускаются и кэшируются в `IndexedDB` с ключом `baseUrl + ':' + chunk_index`. Это предотвращает повторный запрос тех же данных во время шторма повторных попыток.

### 5. Адаптивный пейсинг (Adaptive Pacing)

- Вместо простого снижения параллелизма в плохих сетях TASFA **регулирует темп** отправки чанков, добавляя задержки `setTimeout` в зависимости от качества канала.

| Оценка канала | Пейсинг загрузки | Пейсинг скачивания | Coalesce |
|-----------|---------------|-----------------|----------|
| >= 85     | 0 мс          | 0 мс            | 64       |
| 65 – 84   | 15 мс         | 0 мс            | 48       |
| 45 – 64   | 15 мс         | 10 мс           | 32       |
| < 45      | 35 мс         | 30 мс           | 16       |

- Это предотвращает коллапс TCP-окна на каналах с высокими потерями, а также блокировки из-за раздутия буферов.
- Когда оценка канала падает ниже 45, таймауты увеличиваются (до 12 с), а коэффициент отката при повторных попытках снижается с 1,6 до 1,3.

### 6. Укрепление сессий

- ID загрузки и токены — это случайные шестнадцатеричные строки длиной 16 и 24 байта.
- `flock` предотвращает состояние гонки при обновлении битовой карты из параллельных запросов.
- Сессии скачивания ограничены TTL (`TASFA_DOWNLOAD_TTL = 86400`) и удаляются по истечении срока.

## Стоит ли жертвовать RPS?

По сравнению с простой раздачей статических файлов, TASFA явно снижает **RPS (количество запросов в секунду)**. Однако это намеренный компромисс ради следующих преимуществ:

| Что жертвуем | Что получаем |
|------------------|---------------|
| Дополнительные накладные расходы на шифрование/дешифрование на горячем пути | **Целостность на уровне чанка**: несовпадение тега обнаруживается немедленно; повреждённые чанки никогда не попадают на диск |
| CPU на вычисление HTP-скаляров и валидацию групп | **Раннее обнаружение повреждений**: ошибки, которые хэш всего файла обнаружил бы только в конце, ловятся на уровне группы |
| Файловый I/O битовой карты и блокировки `flock` | **Защита от DoS**: блокирует злоупотребление повторной отправкой одного чанка и атаки с усечёнными файлами |
| Невозможность одновременной загрузки нескольких файлов из-за очереди | **Стабильность соединений браузера**: бесперебойная передача в рамках ограничения в 6 соединений на origin без взаимоблокировок |
| Сниженная скорость отправки чанков из-за пейсинга | **Устойчивость к нестабильным каналам**: непрерывная передача без коллапса TCP-окна в условиях высоких потерь и задержек |
| Потребление памяти и диска для кэширования IndexedDB и состояния сессий | **Эффективность повторных попыток**: после разрыва сети уже полученные чанки не запрашиваются повторно |

### Заключение

TASFA не ставит целью «самую быструю передачу файлов». Вместо этого он спроектирован так, чтобы **файлы доходили до назначения полностью и без повреждений даже в нестабильных сетях и при наличии вредоносных клиентов**.

- **С точки зрения загрузки**: зашифрованный поток чанков с изоляцией сессий + HTP-подсказки + защита от DoS через битовую карту + очередь = стабильная и целостная загрузка.
- **С точки зрения скачивания**: сессия на основе handshake + отслеживание битовой карты + кэширование в IndexedDB + адаптивный пейсинг = бесперебойное скачивание.

Для простого блога TASFA может быть избыточен, но для **сообщества с большим количеством вложений** потеря RPS является оправданной платой за ясные технические преимущества.
