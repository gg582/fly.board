# 내부 개발 및 개선 보고 (2차): 엔진 비동기 스케줄링 지능화 및 GIF WebM 강제 변환

미디어 처리 및 파일 송수신 성능을 극대화하고 동시 연결이 느려지는 현상을 완벽하게 해소하기 위해 엔진 스케줄러와 GIF 처리 방식을 근본적으로 개선했습니다.

---

## 1. 전송 풀 격리 및 연산 스레드 스케줄링 지능화 (스타베이션 해결)

### 문제점
* 파일 업로드 완료 후 미디어(비디오, 썸네일 등) 프리뷰를 생성하는 작업(`upload_finalize_worker`)이 기존에는 파일 다운로드/업로드 청크 I/O를 전담하는 핵심 스레드 풀(`g_tasfa_pool`)에서 같이 구동되었습니다.
* 이로 인해 비디오 인코딩(ffmpeg) 도중 동시 처리 수 한계(`TASFA_MAX_MEDIA_CONCURRENCY`)에 걸리면, 해당 비동기 I/O 워커 스레드가 `pthread_cond_wait`로 대기 상태에 빠져 다른 사용자의 다운로드/업로드 청크 요청을 처리하지 못하고 락에 묶였습니다.
* 동시 업로드가 몰릴 시 I/O 풀 전체의 스레드가 고갈(Thread Starvation)되어 웹 서버의 모든 커넥션이 멈추거나 급격히 느려지는 현상이 유발되었습니다.

### 개선 사항 (지능형 격리 스케줄링)
* **스레드 풀 도메인 분리**: 무겁고 락 대기가 발생하는 파이널라이즈 및 미디어 처리 작업을 가볍고 빨라야 하는 전송 스레드 풀(`g_tasfa_pool`)에서 **완벽히 격리**했습니다.
* **백그라운드 연산 풀 위임**: 미디어 연산 작업을 libttak 백그라운드 스레드 풀(`g_background_pool`)로 위임하여 실행되도록 바꿨습니다.
* **도메인 및 우선순위 최적화**: 작업을 CPU 바운드 연산 도메인인 `TTAK_TASK_DOMAIN_THREAD`로 지정하고, 태스크의 Urgency(우선순위)를 `30`(낮음)으로 세팅했습니다.
* **효과**: 미디어 연산 및 ffmpeg가 아무리 백그라운드에서 CPU를 점유하고 락 대기를 하더라도, 실시간 파일 전송을 처리하는 `g_tasfa_pool`은 단 1ms도 막히지 않고 항시 최대 대역폭으로 동시 청크 전송을 완수할 수 있게 되었습니다.

## 2. GIF 미디어의 WebM 비디오 강제 변환 및 비디오 일원화

### 문제점
* 기존에는 GIF 파일 업로드 시 GIF 원본을 유지하면서 필요할 때만 MP4 프리뷰로 변환하는 방식을 취했으나, 이로 인해 서빙 로직에 분기가 얽히고 비디오 루프와 이미지 썸네일 처리가 혼재되어 비효율적이었습니다.

### 개선 사항
* **업로드 시점 WebM 인코딩**: 업로드 완료 시점에 GIF 포맷(`image/gif`)이 감지되면 원본 임시 파일을 **진짜 WebM 비디오 파일**로 강제 변환하여 `public/uploads/`에 `.webm` 확장자로 저장합니다.
* **DB 메타데이터 완전 일치**: 변환된 웹엠 비디오의 크기를 측정하여 메타데이터의 `total_size`를 갱신하고, 파일명을 `.webm`으로 변경하여 DB(`files` 테이블)에 마임타입 **`video/webm`**으로 영구 등록합니다.
* **일관된 렌더링**: 데이터베이스 수준에서 처음부터 완벽한 비디오 파일로 취급되므로, 썸네일 생성 및 본문 렌더링 시 자동으로 완성도 높은 비디오 파이프라인을 그대로 타게 되며, 불필요했던 GIF 예외 코드가 제거되어 아키텍처가 단순화되었습니다.
