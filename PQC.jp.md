# PQC（耐量子暗号）署名の解析

## 概要

fly.board は投稿に「量子耐性」の署名を付与している。この文書では、その機能が**どのような原理で動作しているか**、そして**実際の限界は何か**を技術的に記述する。

## PQC はどのように実現されているか

### 1. 使用アルゴリズム：ML-DSA-65

- `src/crypto/fly_crypto.h` のコメントおよび `README` による。
- ML-DSA-65 は **NIST 標準化された格子（lattice）ベースの署名方式** である。
- 従来の RSA/ECDSA が整数因数分解・離散対数問題に依存するのに対し、格子問題は現時点で量子コンピュータ（ショアのアルゴリズム）によって多項式時間で解かれることはないと考えられている。

### 2. 実装経路：CWIST フレームワーク → BoringSSL

- `fly_crypto.c` は `<cwist/security/pqc/pqc_sig.h>` を条件付きでインクルードする。
- CWIST フレームワーク内部には BoringSSL が組み込まれており、BoringSSL は `crypto/mldsa/` に ML-DSA の実装を含む。
- したがって fly.board 自身が独自の PQC アルゴリズムを実装したのではなく、**CWIST/BoringSSL が提供する標準 ML-DSA API を呼び出す**形態である。

### 3. 署名対象

- 投稿の作成・更新時に以下の文字列が署名される：
  ```
  title + "\n" + content
  ```
- この署名は SQLite `posts` テーブルの `pqc_signature` カラムに Base64 で保存される。
- 投稿詳細表示時に `fly_crypto_verify()` で検証され、UI に "verified" バッジが表示される。

### 4. 鍵管理

- `fly_crypto_init()` は**単一のグローバル署名鍵**を生成し、メモリ上に保持する。
- 鍵はアプリケーション起動時に一度生成され、終了時に `fly_crypto_cleanup()` で解放される。

## 限界と注意事項

### 1. 著者認証ではなく「サーバー保証」

- 鍵は**サーバーあたり 1 つ**であり、ユーザーごとの分離は存在しない。
- したがってこの署名は「どのユーザーが書いたか」を証明するものではなく、**「このサーバーがこの投稿を承認・保証する」**という意味に近い。
- サーバーが侵害されると、攻撃者はサーバーの秘密鍵を使って任意の投稿に有効な署名を生成できる。

### 2. 条件付きコンパイルによる移植性の落とし穴

- `fly_crypto.c` は `#ifdef HAVE_PQC` で囲まれている。
- ビルド環境に `<cwist/security/pqc/pqc_sig.h>` がない場合、すべての暗号関数は **no-op（ダミー）** となり、`fly_crypto_sign()` は `false` を、`fly_crypto_verify()` は常に失敗する。
- つまり「PQC をサポートする」と常に保証されるわけではなく、実際には **CWIST PQC モジュールの有無に完全に依存**する。

### 3. 署名範囲の限界

- 署名対象は `title` と `content` のみである。
- 作成者 ID、作成時刻、掲示板情報、添付ファイルのメタデータなどは**署名に含まれない**。
- 攻撃者が投稿本文はそのままに作成者情報だけを書き換えた場合、署名は依然として有効なまま残る。

### 4. 伝送層は依然として従来型暗号

- fly.board は HTTP/3（QUIC）と TLS を使用する。
- TLS ハンドシェイクおよびセッション暗号化は**依然として ECC（RSA）ベース**である。
- PQC 署名は「投稿本文の完全性」にのみ適用され、**クライアント・サーバー間のすべての通信が量子安全であるわけではない**。

### 5. 鍵ローテーションおよび失効戦略の欠如

- グローバル鍵は一度生成されると再起動まで維持される。
- 鍵が漏洩した場合、過去のすべての投稿の署名信頼度が一度に崩壊する。
- 鍵ローテーション、分散保存（HSM）、失効（revocation）メカニズムが存在しない。

## 結論

fly.board の PQC 署名は、**NIST 標準 ML-DSA-65 を通じて投稿本文の完全性を保証する**機能である。ブログエンジンとしては異例の試みであり、「量子時代においても本文の改ざんが検出可能である」という意味を持つ。

しかし、**著者ごとの非対称認証、メタデータ保護、伝送層の PQC、鍵ライフサイクル管理**などが欠けているため、マーケティング的に「量子攻撃にも耐える」と謳うには限界が明確である。この機能は「PQC 署名デモ」程度に理解するのが適切である。
