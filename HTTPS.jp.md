# HTTP/1.1・HTTP/2・HTTP/3 同時支援の解析

## 概要

fly.board は `cwist_app_use_https2()` と `cwist_app_use_https3()` を同時に有効化し、すべてのレスポンスに `Alt-Svc: h3=":%d"; ma=86400, h2=":%d"; ma=86400` ヘッダーを含める。これは **HTTP/3 を優先し、不可能な場合は HTTP/2、さらにそれも不可なら HTTP/1.1 へ自動フォールバック** する構造である。

## なぜ「性能を犠牲にしても」3つすべてを支援するのか

### 1. HTTP/3（QUIC）— 理想的だが環境制約が大きい

- **利点**: 0-RTT ハンドシェイク、TCP Head-of-Line Blocking の完全排除、接続マイグレーション（ネットワーク切り替え時もセッション維持）。
- **性能犠牲要因**: UDP ベースの QUIC は追加の暗号化・パケット再構成・輻輳制御オーバーヘッドにより、**HTTP/2 に比べ CPU 使用量が高い**。また CWIST 内部の BoringSSL + ngtcp2/nghttp3 スタックは、純粋な HTTP/2 TCP スタックより多くのメモリとコードパスを占有する。
- **支援理由**: 最新ブラウザ（Chrome、Firefox、Safari）は HTTP/3 を優先するが、**エンタープライズファイアウォール、一部 NAT、旧式プロキシ、モバイルキャリア級ネットワークは UDP をブロックまたは QoS 制限**する。HTTP/3 のみを支援すると、このような環境のユーザーは接続自体が不可能になる。

### 2. HTTP/2（TLS 1.3）— 現在の「安定的な最大公約数」

- **利点**: 単一 TCP 接続上のマルチプレクシングで HTTP/1.1 の接続浪費を解決。サーバープッシュ（選択的）、ヘッダー圧縮（HPACK）。
- **性能犠牲要因**: HTTP/2 は HTTP/1.1 より状態機械が複雑で、ストリーム優先度・フロー制御（window update）処理に CPU サイクルを消費する。ベンチマーク上 HTTP/2 は理論上 HTTP/1.1 Keep-Alive より高速だが、**fly.board のベンチマークは HTTP/2 で測定**されたと明示している — これは HTTP/3 がまだすべてのベンチマークツールとネットワーク環境で予測可能な結果を出せないことを示唆している。
- **支援理由**: HTTP/3 が塞がれる環境での **主力代替手段（primary fallback）**。また `h2load` などの性能測定ツールが HTTP/2 を標準で支援し、デバッグとモニタリングのエコシステムが HTTP/3 より成熟している。

### 3. HTTP/1.1 — 「最後の保険」であり互換性の基盤

- **利点**: 単純さ。テキストベースのプロトコルでデバッグが容易であり、すべてのロードバランサー・リバースプロキシ・ヘルスチェックツールが完璧に支援する。
- **性能犠牲要因**: 単一接続あたり単一リクエスト・レスポンス（パイプラインは事実上失敗）。接続数が増えるとメモリとファイルディスクリプタ消費が急増。HTTP/2 のマルチプレクシングと比較して **並行効率が著しく低い**。
- **支援理由**:
  - **レガシークライアント**: 旧式 CLI ツール、一部組み込み機器、古いクローラーは HTTP/1.1 しか理解できない。
  - **インフラ互換性**: 多くのロードバランサーや WAF は HTTP/2 以上をエンドツーエンドで処理できず、HTTP/1.1 にフォールダウンする。
  - **ヘルスチェックおよびモニタリング**: `curl`、`wget`、Nagios/Zabbix ベースのチェックは依然として HTTP/1.1 が最も安定している。

## Alt-Svc ベースの優先順位交渉

```c
// src/handlers/handlers.c
snprintf(altsvc, sizeof(altsvc), 
    "h3=\":%d\"; ma=86400, h2=\":%d\"; ma=86400", 
    g_config.port, g_config.port);
```

- クライアントが HTTP/2（または HTTP/1.1）で最初の接続を確立すると、サーバーはレスポンスヘッダーで「同じポートの UDP で HTTP/3 も可能だ」と通知する。
- クライアントはこのヒントをキャッシュ（`ma=86400`、24 時間）し、**次のリクエストから HTTP/3 を試行**する。
- HTTP/3 が失敗した場合、クライアントライブラリ（ブラウザなど）は自動的に HTTP/2 → HTTP/1.1 に低下する。

## 性能を犠牲にする代わりに得るもの

| 犠牲にするもの | 得るもの |
|------------------|---------------|
| HTTP/3 単独運用時の理論上最も低いレイテンシ | **全世界ユーザーのアクセシビリティ**: UDP ブロック環境でもサービス可能 |
| HTTP/2 単独運用時の安定的スループット | **将来対応**: HTTP/3 が標準になれば自動的に最適経路に転換 |
| HTTP/1.1 単独運用時の極度の単純さと低メモリ使用 | **TASFA 大容量ファイル転送の最適化**: HTTP/2 マルチプレクシング + HTTP/3 ストリームがチャンク転送に有利 |
| サーバーコード複雑度の増加（3 スタック同時維持） | **運用安定性**: あるプロトコルスタックにバグが生じても、他のスタックで自動迂回 |

## 結論

fly.board が HTTP/1.1・HTTP/2・HTTP/3 をすべて支援するのは、「最新プロトコル一つだけにこだわって最高性能を出す」戦略ではない。むしろ **「どのクライアント、どのネットワーク、どのインフラ環境でも接続可能で、可能な限り最善のプロトコルに自動切換えする」** という実用的な布石である。

これはブログエンジンというよりは、**多様な端末とネットワーク条件を経験するコミュニティプラットフォーム**に適した設計選択である。HTTP/3 の高性能を放棄したのではなく、HTTP/3 が不可能な瞬間に備えて HTTP/2 と HTTP/1.1 という安全網を維持しているのである。
