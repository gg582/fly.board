# HTTP/1.1·HTTP/2·HTTP/3 同时支持分析

## 概述

fly.board 同时启用了 `cwist_app_use_https2()` 与 `cwist_app_use_https3()`，并在所有响应中注入 `Alt-Svc: h3=":%d"; ma=86400, h2=":%d"; ma=86400` 头。这构成了一个**优先使用 HTTP/3，不可用时回退到 HTTP/2，再不可用时进一步降级到 HTTP/1.1**的架构。

## 为何"即使牺牲性能"也要同时支持三种协议

### 1. HTTP/3 (QUIC) — 理想但受环境制约

- **优点**：0-RTT 握手、彻底消除 TCP 队头阻塞、连接迁移（网络切换时仍保持会话）。
- **性能牺牲点**：基于 UDP 的 QUIC 引入了额外的加密、报文重组与拥塞控制开销，**CPU 占用高于 HTTP/2**。此外，CWIST 内部的 BoringSSL + ngtcp2/nghttp3 协议栈比纯 HTTP/2 TCP 栈占用更多内存与代码路径。
- **保留原因**：现代浏览器优先使用 HTTP/3，但**企业防火墙、部分 NAT、旧式代理、移动运营商网络会阻断或对 UDP 进行 QoS 限制**。如果只支持 HTTP/3，这些环境下的用户将无法访问服务。

### 2. HTTP/2 (TLS 1.3) — 当下稳定的"最大公约数"

- **优点**：在单一 TCP 连接上多路复用，解决了 HTTP/1.1 的连接浪费问题；可选的服务器推送；头部压缩 (HPACK)。
- **性能牺牲点**：HTTP/2 的状态机比 HTTP/1.1 更复杂，流优先级与流量控制 (window update) 处理会消耗 CPU 周期。值得注意的是，**fly.board 的基准测试是在 HTTP/2 上测得的**——这暗示 HTTP/3 尚未能在所有基准测试工具和网络环境中给出可预测、可复现的结果。
- **保留原因**：当 HTTP/3 被阻断时的**主要回退方案 (primary fallback)**。此外，`h2load` 等性能测试工具以 HTTP/2 为标准，调试与监控生态也比 HTTP/3 更成熟。

### 3. HTTP/1.1 — "最后手段"与兼容性基线

- **优点**：简单。基于文本的协议易于调试；所有负载均衡器、反向代理和健康检查工具都完美支持。
- **性能牺牲点**：单连接单次请求-响应（流水线实际上已失败）。连接数增加时，内存与文件描述符消耗急剧上升。**并发效率远低于 HTTP/2 的多路复用**。
- **保留原因**：
  - **旧版客户端**：旧版 CLI 工具、部分嵌入式设备、老旧爬虫仅支持 HTTP/1.1。
  - **基础设施兼容性**：许多负载均衡器和 WAF 无法在端侧原生处理 HTTP/2，会降级到 HTTP/1.1。
  - **健康检查与监控**：`curl`、`wget` 以及基于 Nagios/Zabbix 的探针在 HTTP/1.1 上最为稳定。

## 基于 Alt-Svc 的优先级协商

```c
// src/handlers/handlers.c
snprintf(altsvc, sizeof(altsvc), 
    "h3=\":%d\"; ma=86400, h2=\":%d\"; ma=86400", 
    g_config.port, g_config.port);
```

- 客户端通过 HTTP/2（或 HTTP/1.1）建立首次连接时，服务器通过响应头告知"同一端口的 UDP 也支持 HTTP/3"。
- 客户端缓存此提示 (`ma=86400`，24 小时)，**后续请求优先尝试 HTTP/3**。
- HTTP/3 失败时，客户端库（浏览器等）自动降级到 HTTP/2 → HTTP/1.1。

## "牺牲性能"换来的收益

| 牺牲的内容 | 获得的收益 |
|------------------|---------------|
| 仅运行 HTTP/3 时理论最低的延迟 | **全球用户的可访问性**：即使在阻断 UDP 的环境中服务依然可用 |
| 仅运行 HTTP/2 时稳定的吞吐量 | **面向未来**：当 HTTP/3 成为标准后，客户端自动切换到最优路径 |
| 仅运行 HTTP/1.1 时极致的简单与低内存占用 | **TASFA 大文件传输优化**：HTTP/2 多路复用 + HTTP/3 流对分块传输更有利 |
| 降低服务端代码复杂度（只维护一套协议栈） | **运营韧性**：某一协议栈出现 bug 时，流量自动绕行到其他栈 |

## 结论

同时支持 HTTP/1.1、HTTP/2 和 HTTP/3，并不是"死守单一现代协议以换取峰值性能"的策略。相反，这是一种务实的押注：**"无论客户端、网络或基础设施环境如何，服务始终可达，并尽可能自动切换到最佳协议"**。

这一设计选择对于需要面对**多样化终端与网络条件的社区平台**而言，远比单纯的博客引擎更合适。fly.board 并非放弃了 HTTP/3 的高性能，而是在 HTTP/3 不可用的时刻，保留了 HTTP/2 与 HTTP/1.1 作为安全网。
