# TASFA 文件传输协议分析

## TASFA 是什么

TASFA 是 fly.board 使用的 **基于 HTTP/XHR 的文件上传/下载协议**。它不仅仅是文件管道，而是一个集成了分块加密、完整性校验、不稳定网络应对与 DoS 防护的协议层。

## 核心构成要素

### 1. 分块 AES-256-GCM 流式加密

- 每个上传会话生成唯一的 `stream_key` 与 `iv_seed`。
- 每个数据块（chunk）使用 `AES-256-GCM` 加密，并以 `upload_id:chunk_index` 作为 AAD（附加认证数据）。
- 标签（tag）验证失败时，该数据块会被立即丢弃。
- 服务器收到密文后解密，并将明文直接写入预分配的临时文件偏移量 `chunk_index * chunk_size` 处。
- 此加密确保了**会话间独立性**：某一会话的密钥泄露不会影响其他会话。但这并非替代 TLS，而是会话级别的附加完整性校验层。

### 2. 6 槽 HTP（Hexagonal Tortoise Problem）本地修复 Lattice

- HTP **不是传输协议**，而是**本地修复提示（hint）**。
- 数据块按每 6 个一组进行分组。
- 每个数据块的 SHA-512 前 8 字节被解释为 64 位整数 `H`，计算 `raw_scalar = H % modulus_M`。
- 完整组需满足三条直线不变式（`L1 == L2 == L3`）；若不满足，仅调整 `v3` 与 `v5` 使其以最小增量达到平衡。
- 上传完成时，服务器逐组验证不变式。不匹配则返回 `400 Bad Request`。
- 这提供了一种轻量级早期检测手段，可在**数据块级别的替换或损坏**尚未累积到整文件哈希失效前即发现异常。

### 3. 文件级顺序队列（浏览器连接稳定性）

- **同一时间仅上传一个文件**。
- 多文件选择时，文件进入 `FileUploadQueue` 排队，只有当前文件成功或失败后，下一个文件才开始传输。
- 此举尊重浏览器对同一源（origin）6 个并发连接的限制，防止连接池耗尽导致的死锁。

### 4. 二进制位图用于 DoS 缓解与状态追踪

- 上传进度以**稠密二进制位图**形式保存在 `state.bin` 中（每个数据块 1 字节，`'0'`/`'1'`）。
- 已标记为 `'1'` 的数据块索引若再次收到重传，服务器直接拒绝。由此阻止通过重复发送同一块来消耗磁盘 I/O 的 DoS 攻击。
- 使用 `pwrite(..., 1, chunk_index)` 实现 O(1) 原子写入，热路径上**不存在 JSON 解析**。
- 下载侧同样维护位图，已获取的数据块跳过，并缓存至以 `baseUrl + ':' + chunk_index` 为键的 `IndexedDB` 中。重试风暴（retry storm）期间不会重复请求相同数据。

### 5. 自适应步速（Adaptive Pacing）

- 网络状况恶化时，TASFA 并非简单降低并行度，而是根据链路质量在数据块调度之间注入 `setTimeout` 延迟（pacing）。

| 链路评分 | 上传 pacing | 下载 pacing | Coalesce |
|-----------|---------------|-----------------|----------|
| >= 85     | 0 ms          | 0 ms            | 64       |
| 65 – 84   | 15 ms         | 0 ms            | 48       |
| 45 – 64   | 15 ms         | 10 ms           | 32       |
| < 45      | 35 ms         | 30 ms           | 16       |

- 防止高丢包链路下 TCP 窗口崩溃，避免缓冲区膨胀（buffer-bloat）导致的传输停滞。
- 链路评分低于 45 时，超时时间延长（最高 12 秒），重试退避系数从 1.6 降至 1.3。

### 6. 会话加固

- 上传 ID 与 Token 分别为 16 字节与 24 字节随机十六进制字符串。
- 使用 `flock` 防止并发请求间的位图更新竞争。
- 下载会话受 TTL 限制（`TASFA_DOWNLOAD_TTL = 86400`），过期后自动丢弃。

## RPS 牺牲是否值得

> **注意**：以下 RPS 权衡**仅适用于文件传输端点**（`/file/upload`、`/file/download` 等）。基准测试中测得的首页、板块列表等页面 RPS 不受 TASFA 影响。

与简单的静态文件服务相比，TASFA 显然会带来 **RPS（每秒请求数）损失**。然而，这种损失是为了换取以下价值而有意做出的权衡：

| 牺牲的内容 | 获得的内容 |
|------------------|---------------|
| 热路径上的额外加解密开销 | **分块完整性**：标签不匹配立即被检测到，损坏的数据块不会落盘 |
| HTP 标量计算与分组验证消耗的 CPU | **早期损坏检测**：整文件哈希只能在传输结束后才能发现的错误，可在分组级别提前捕获 |
| 位图文件 I/O 与 `flock` 会话锁 | **DoS 防护**：阻止重复发送同一块的滥用，以及截断文件攻击（truncated-file attack） |
| 顺序队列导致无法同时上传多个文件 | **浏览器连接稳定性**：在单个 origin 的 6 连接限制内实现无死锁的稳定传输 |
| pacing 带来的较低数据块发送速率 | **不稳定链路恢复力**：在高丢包/高延迟环境下仍能持续传输，避免 TCP 窗口崩溃 |
| IndexedDB 缓存与会话状态占用的内存与磁盘 | **重试效率化**：网络断开后重新连接时，已接收的数据块不会被重复请求 |

### 结论

TASFA 并不追求"最快的文件传输"。相反，它的设计目标是：**即使在不稳定网络和恶意客户端环境下，文件也能完整、无损地到达目的地**。

- **上传视角**：会话级加密数据流 + HTP 提示 + 位图 DoS 防护 + 顺序队列 = 稳定且无损的上传。
- **下载视角**：基于握手的会话 + 位图追踪 + IndexedDB 缓存 + 自适应步速 = 不间断的下载。

对于纯博客而言 TASFA 可能过于复杂；但对于**以附件为核心的社区论坛**，这种 RPS 损失具备明确的技术合理性，值得承担。
