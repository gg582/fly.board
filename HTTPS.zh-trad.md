# HTTP/1.1·HTTP/2·HTTP/3 同時支援分析

## 概述

fly.board 同時啟用了 `cwist_app_use_https2()` 與 `cwist_app_use_https3()`，並在所有回應中注入 `Alt-Svc: h3=":%d"; ma=86400, h2=":%d"; ma=86400` 標頭。這構成了一個**優先使用 HTTP/3，不可用時回退到 HTTP/2，再不可用時進一步降級到 HTTP/1.1**的架構。

## 為何「即使犧牲效能」也要同時支援三種協定

### 1. HTTP/3 (QUIC) — 理想但受環境制約

- **優點**：0-RTT 交握、徹底消除 TCP 佇列頭阻塞、連線遷移（網路切換時仍保持工作階段）。
- **效能犧牲點**：基於 UDP 的 QUIC 引入了額外的加密、封包重組與壅塞控制開銷，**CPU 佔用高於 HTTP/2**。此外，CWIST 內部的 BoringSSL + ngtcp2/nghttp3 協定棧比純 HTTP/2 TCP 棧佔用更多記憶體與程式碼路徑。
- **保留原因**：現代瀏覽器優先使用 HTTP/3，但**企業防火牆、部分 NAT、舊式代理、行動電信商網路會阻斷或對 UDP 進行 QoS 限制**。如果只支援 HTTP/3，這些環境下的使用者將無法存取服務。

### 2. HTTP/2 (TLS 1.3) — 當下穩定的「最大公約數」

- **優點**：在單一 TCP 連線上多工，解決了 HTTP/1.1 的連線浪費問題；可選的伺服器推送；標頭壓縮 (HPACK)。
- **效能犧牲點**：HTTP/2 的狀態機比 HTTP/1.1 更複雜，串流優先順序與流量控制 (window update) 處理會消耗 CPU 週期。值得注意的是，**fly.board 的基準測試是在 HTTP/2 上測得的**——這暗示 HTTP/3 尚未能在所有基準測試工具和網路環境中給出可預測、可複現的結果。
- **保留原因**：當 HTTP/3 被阻斷時的**主要回退方案 (primary fallback)**。此外，`h2load` 等效能測試工具以 HTTP/2 為標準，偵錯與監控生態也比 HTTP/3 更成熟。

### 3. HTTP/1.1 — 「最後手段」與相容性基線

- **優點**：簡單。基於文字協定易於偵錯；所有負載平衡器、反向代理和健康檢查工具都完美支援。
- **效能犧牲點**：單連線單次請求-回應（管線化實際上已失敗）。連線數增加時，記憶體與檔案描述元消耗急劇上升。**並行效率遠低於 HTTP/2 的多工**。
- **保留原因**：
  - **舊版用戶端**：舊版 CLI 工具、部分嵌入式裝置、老舊爬蟲僅支援 HTTP/1.1。
  - **基礎設施相容性**：許多負載平衡器和 WAF 無法在端側原生處理 HTTP/2，會降級到 HTTP/1.1。
  - **健康檢查與監控**：`curl`、`wget` 以及基於 Nagios/Zabbix 的探針在 HTTP/1.1 上最為穩定。

## 基於 Alt-Svc 的優先順位協商

```c
// src/handlers/handlers.c
snprintf(altsvc, sizeof(altsvc), 
    "h3=\":%d\"; ma=86400, h2=\":%d\"; ma=86400", 
    g_config.port, g_config.port);
```

- 用戶端透過 HTTP/2（或 HTTP/1.1）建立首次連線時，伺服器透過回應標頭告知「同一連接埠的 UDP 也支援 HTTP/3」。
- 用戶端快取此提示 (`ma=86400`，24 小時)，**後續請求優先嘗試 HTTP/3**。
- HTTP/3 失敗時，用戶端程式庫（瀏覽器等）自動降級到 HTTP/2 → HTTP/1.1。

## 「犧牲效能」換來的收益

| 犧牲的內容 | 獲得的收益 |
|------------------|---------------|
| 僅執行 HTTP/3 時理論最低的延遲 | **全球使用者的可存取性**：即使在阻斷 UDP 的環境中服務依然可用 |
| 僅執行 HTTP/2 時穩定的吞吐量 | **面向未來**：當 HTTP/3 成為標準後，用戶端自動切換到最優路徑 |
| 僅執行 HTTP/1.1 時極致的簡單與低記憶體佔用 | **TASFA 大檔案傳輸最佳化**：HTTP/2 多工 + HTTP/3 串流對分塊傳輸更有利 |
| 降低伺服器端程式碼複雜度（只維護一套協定棧） | **營運韌性**：某一協定棧出現 bug 時，流量自動繞行到其他棧 |

## 結論

同時支援 HTTP/1.1、HTTP/2 和 HTTP/3，並不是「死守單一現代協定以換取峰值效能」的策略。相反，這是一種務實的押注：**「無論用戶端、網路或基礎設施環境如何，服務始終可達，並盡可能自動切換到最佳協定」**。

這一設計選擇對於需要面對**多樣化終端與網路條件的社群平台**而言，遠比單純的部落格引擎更合適。fly.board 並非放棄了 HTTP/3 的高效能，而是在 HTTP/3 不可用的時刻，保留了 HTTP/2 與 HTTP/1.1 作為安全網。
