# PQC（後量子密碼）簽名分析

## 概述

fly.board 在文章上標註了「量子抗性」簽名。本文件說明該功能**實際工作原理**以及**真正的局限性**。

## 如何實現 PQC

### 1. 使用演算法：ML-DSA-65

- 根據 `src/crypto/fly_crypto.h` 的註釋及 `README` 描述。
- ML-DSA-65 是 **NIST 標準化的基於格（lattice）的簽名演算法**。
- 傳統 RSA/ECDSA 依賴整數分解與離散對數問題，而這些問題可被量子電腦上的秀爾演算法（Shor's algorithm）破解；格問題目前被認為在多項式時間內無法被量子演算法求解。

### 2. 實作路徑：CWIST 框架 → BoringSSL

- `fly_crypto.c` 條件包含 `<cwist/security/pqc/pqc_sig.h>`。
- CWIST 框架內部嵌入了 BoringSSL，BoringSSL 在 `crypto/mldsa/` 中包含 ML-DSA 實作。
- 因此 fly.board 並未自行實作獨立的 PQC 演算法，而是**呼叫 CWIST/BoringSSL 提供的標準 ML-DSA API**。

### 3. 簽名對象

- 文章建立或修改時，對以下字串進行簽名：
  ```
  title + "\n" + content
  ```
- 簽名以 Base64 形式儲存在 SQLite `posts` 表的 `pqc_signature` 欄位中。
- 查看文章詳情時，透過 `fly_crypto_verify()` 驗證簽名，並在 UI 上顯示 "verified" 徽章。

### 4. 金鑰管理

- `fly_crypto_init()` 生成**單一全域簽名金鑰**，並保留在記憶體中。
- 金鑰在應用程式啟動時生成一次，退出時由 `fly_crypto_cleanup()` 釋放。

## 局限與注意事項

### 1. 並非作者認證，而是「伺服器背書」

- 金鑰為**每伺服器一個**，沒有按使用者隔離。
- 因此該簽名不能證明「哪個使用者撰寫」，而更接近於**「該伺服器認可並擔保此文章」**的含義。
- 一旦伺服器被攻破，攻擊者即可利用伺服器私鑰為任意文章生成有效簽名。

### 2. 條件編譯帶來的可移植性陷阱

- `fly_crypto.c` 被 `#ifdef HAVE_PQC` 包裹。
- 若建置環境中不存在 `<cwist/security/pqc/pqc_sig.h>`，所有密碼函式都會退化為 **no-op（空殼）**：`fly_crypto_sign()` 回傳 `false`，`fly_crypto_verify()` 永遠失敗。
- 換言之，「支援 PQC」並非恆定保證，實際**完全依賴 CWIST PQC 模組是否存在**。

### 3. 簽名範圍有限

- 僅對 `title` 與 `content` 進行簽名。
- 作者 ID、建立時間、板塊資訊、附件元資料等**未被納入簽名**。
- 攻擊者可在保持正文不變的情況下竄改作者資訊，而簽名仍然有效。

### 4. 傳輸層仍使用傳統加密

- fly.board 使用 HTTP/3（QUIC）與 TLS。
- TLS 交握及會話加密**仍然基於 ECC/RSA**。
- PQC 簽名僅適用於「文章正文完整性」，並不意味著**用戶端與伺服器之間的全部通訊都是量子安全的**。

### 5. 缺乏金鑰輪換與撤銷策略

- 全域金鑰一旦生成，在重啟前不會輪換。
- 若金鑰洩露，所有歷史文章簽名的可信度將同時崩潰。
- 不存在金鑰輪換（rotation）、分散式儲存（HSM）、撤銷（revocation）機制。

## 結論

fly.board 的 PQC 簽名透過 **NIST 標準 ML-DSA-65 保障文章正文完整性**。對於部落格引擎而言，這是一種罕見且有意義的嘗試，意味著即使在量子計算時代，正文篡改仍可被偵測。

然而，由於缺少**按作者的非對稱認證、元資料保護、傳輸層 PQC 以及金鑰生命週期管理**，將其行銷為「可抵禦量子攻擊」存在明顯誇大。將此功能理解為「PQC 簽名展示」更為恰當。
